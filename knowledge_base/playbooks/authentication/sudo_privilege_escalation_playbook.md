# Sudo Privilege Escalation Playbook

## Purpose

This playbook supports investigation of suspicious sudo activity and privileged command execution on Linux hosts.

Use it to determine whether sudo activity is expected administration, automation, post-login escalation, or evidence of account compromise.

## When to Use

- Wazuh reports sudo execution by an unexpected user.
- A privileged command runs shortly after suspicious authentication.
- Sudo activity occurs outside approved maintenance windows.
- A command modifies users, packages, services, firewall rules, SSH configuration, or security controls.
- A non-standard account runs administrative commands.

## Detection Signals

- `sudo` command execution by a user not normally associated with administration.
- Repeated denied sudo attempts.
- Sudo command following SSH failures or an unusual successful login.
- Commands such as `useradd`, `usermod`, `passwd`, `chmod`, `chown`, `systemctl`, `crontab`, package managers, shells, or network tools.
- Sudo to root shell through `sudo su`, `sudo -i`, or equivalent.
- Sudo activity on production or internet-facing host without change reference.

## Initial Triage

- Identify user, host, command, working directory, timestamp, and target user.
- Determine whether the user is approved for sudo on this host.
- Check whether the activity matches a change ticket, maintenance window, automation job, or deployment pipeline.
- Review the authentication timeline before sudo execution.
- Check whether the sudo command was allowed or denied.
- Assess whether the command changes system state or only reads information.

## Evidence to Collect

- Raw sudo log event with command line, user, target user, and host.
- Authentication events for the same user and source before the sudo command.
- Shell command context where available.
- File integrity events for modified files.
- Package manager logs, systemd changes, cron changes, user/group changes, and SSH key changes.
- Owner confirmation or change record for the privileged command.
- Network and DNS activity after the privileged action.

## Investigation Steps

1. Classify the command as read-only, configuration change, identity change, persistence-related, network-related, or destructive.
2. Confirm whether the user normally performs this command on this host.
3. Validate the command against approved maintenance or deployment records.
4. Review prior SSH or local login anomalies for the same user.
5. Check for package installation, service creation, file permission changes, or new scheduled tasks after the sudo command.
6. Review whether the command disables security controls, modifies logging, changes firewall policy, or creates accounts.
7. Compare the command path and arguments with known administration scripts.
8. Determine whether the activity is authorized, suspicious, or likely unauthorized escalation.

## Correlation Checks

- Correlate with SSH success after failures.
- Correlate with suspicious package activity.
- Correlate with new or modified systemd service files.
- Correlate with file writes to `/etc`, `/usr/local/bin`, `/tmp`, `/var/tmp`, and user home directories.
- Correlate with outbound network activity, DNS beaconing, or Suricata alerts.
- Correlate with previous sudo events by the same user and host.

## False Positive Conditions

- Command is linked to approved maintenance.
- User is an approved administrator and command matches their normal duties.
- Command is executed by a known automation platform.
- Change was performed through an approved deployment pipeline.
- Command is read-only and expected for troubleshooting.
- Denied sudo attempts are explained by user error and no privileged action succeeded.

## Escalation Criteria

- Sudo follows suspicious SSH login or account compromise indicators.
- Command creates users, modifies sudoers, changes SSH keys, or disables security tooling.
- Command starts a root shell or executes a script from a temporary location.
- User is not approved for administrative access.
- Command occurs outside maintenance and owner cannot confirm it.
- Sudo is followed by persistence, package installation, or suspicious outbound traffic.

## Containment Actions

- Preserve sudo logs and related command context before account or host changes.
- Temporarily suspend the account if unauthorized privileged use is likely and approval is obtained.
- Isolate the host if sudo activity created persistence or launched suspicious binaries.
- Revoke active sessions tied to the user when account compromise is suspected.
- Avoid removing files before preserving evidence.

## Remediation Actions

- Rotate credentials and SSH keys for the user when compromise cannot be excluded.
- Review and correct sudoers configuration.
- Remove unauthorized users, groups, keys, services, packages, or scheduled tasks.
- Restore modified security controls and logging configuration.
- Add detection or policy controls for unauthorized sudo paths or commands.
- Document required access review with system owner.

## Closure Criteria

- Command purpose and authorization are documented.
- Prior authentication and post-command activity were reviewed.
- Related package, service, file, process, and network signals were checked.
- Any unauthorized changes were remediated or tracked.
- Owner confirmation or compromise response is recorded.
- Residual risk and follow-up access review are documented.

## Analyst Notes

- Sudo is context-sensitive; the same command may be benign for one user and critical for another.
- Sudo after suspicious login should be treated as possible compromise until proven otherwise.
- Commands that create persistence require host-level review.
- Do not classify as false positive solely because the command is common.
