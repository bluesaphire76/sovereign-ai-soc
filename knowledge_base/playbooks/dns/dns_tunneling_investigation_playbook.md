# DNS Tunneling Investigation Playbook

## Purpose

This playbook supports investigation of DNS behavior that may indicate tunneling, data exfiltration, or covert command-and-control.

It focuses on high volume queries, long labels, entropy, unusual query types, parent-domain repetition, and host/network correlation.

## When to Use

- A host sends unusually high DNS query volume.
- Queries contain long, random-looking, or high-entropy subdomains.
- Query types include suspicious TXT, NULL, or uncommon records.
- Many unique subdomains share the same parent domain.
- DNS volume, byte count, or frequency is abnormal for the host role.
- DNS behavior aligns with host compromise or Suricata alerts.

## Detection Signals

- Long labels or deeply nested subdomains.
- High ratio of unique subdomains to parent domain.
- Repeated TXT, NULL, CNAME, or uncommon DNS query types.
- High query count from one host to one parent domain.
- DNS responses with unusual sizes, failures, or repeated NXDOMAIN.
- Queries that appear encoded, compressed, or random.
- DNS activity followed by suspicious outbound traffic.

## Initial Triage

- Identify requesting host, user context, parent domain, query count, query types, and time window.
- Review sample queried names for length, entropy, encoded strings, or repeated patterns.
- Determine whether the parent domain is approved, internal, vendor-owned, or suspicious.
- Compare current DNS volume to the host baseline.
- Check whether DNS resolver is internal, approved, or bypassed.
- Review whether the host has recent suspicious login, sudo, package, process, or service alerts.

## Evidence to Collect

- DNS query samples including full query name, type, response code, and timestamp.
- Query volume by host, parent domain, and query type.
- Unique subdomain count and label length distribution where available.
- Resolver path and whether external resolvers were used.
- Network flow metadata for resolved IPs and external DNS servers.
- Host telemetry for process, service, package, and authentication anomalies.
- Domain reputation, ownership, registration, and business validation.

## Investigation Steps

1. Group queries by parent domain and requesting host.
2. Calculate query rate, unique subdomain count, and query type distribution.
3. Review sample labels for base64-like, hex-like, or random-looking strings.
4. Check whether the behavior matches known CDN, SaaS, telemetry, EDR, or browser patterns.
5. Validate whether the application owner recognizes the domain and query volume.
6. Review whether the host bypasses approved resolvers or sends direct external DNS.
7. Correlate with Suricata alerts, outbound flows, and endpoint events.
8. Determine whether the evidence supports benign application behavior, suspicious tunneling, or likely exfiltration.

## Correlation Checks

- Correlate with DNS C2 beaconing patterns.
- Correlate with Suricata high severity alerts involving DNS, suspicious protocol, or exfiltration.
- Correlate with host process execution and suspicious binaries.
- Correlate with package installation or systemd service creation before DNS spike.
- Correlate with authentication anomalies and privilege escalation.
- Correlate with proxy, firewall, and flow data for external communication.

## False Positive Conditions

- Queries belong to approved CDN, SaaS, browser safe browsing, telemetry, EDR, monitoring, or security tool.
- Parent domain is owned by an approved vendor and behavior matches documentation.
- High query volume is expected for the application role.
- Query strings are long but generated by known business software.
- Owner validates the application and timing.
- No host or network compromise indicators are present.

## Escalation Criteria

- DNS labels are high entropy and unexplained.
- Query volume or byte volume is materially abnormal for the host.
- TXT, NULL, or unusual query types are frequent and unexplained.
- Host has suspicious process, service, package, or authentication events.
- Parent domain is newly registered, rare, or linked to suspicious infrastructure.
- DNS activity may contain encoded data or align with exfiltration timeline.

## Containment Actions

- Preserve DNS samples, volume metrics, host telemetry, and flow evidence.
- Block parent domain or external resolver path only after validation and approval.
- Isolate host when tunneling correlates with compromise or exfiltration indicators.
- Increase DNS logging for the host and parent domain.
- Avoid deleting local artifacts before process and service evidence is captured.

## Remediation Actions

- Remove tunneling tool, malware, or unauthorized software if confirmed.
- Restore approved DNS resolver configuration.
- Restrict direct external DNS egress where policy supports it.
- Review credentials and secrets on the affected host if exfiltration is possible.
- Add detection for parent domain, query type, label length, or volume threshold.
- Conduct data exposure review when exfiltration is plausible.

## Closure Criteria

- Parent domain ownership and business purpose are validated or suspicious classification is documented.
- Query volume, query types, label patterns, and resolver behavior are reviewed.
- Host and network correlations were checked.
- Exfiltration risk is assessed and documented.
- Any blocking, isolation, or cleanup action is approved and audited.
- Residual risk and follow-up monitoring are documented.

## Analyst Notes

- DNS tunneling investigations require samples, volume, and host context.
- Long DNS labels are not enough; validate application behavior.
- Direct external DNS from managed endpoints is a stronger signal.
- Escalate quickly when suspicious DNS aligns with host compromise.
