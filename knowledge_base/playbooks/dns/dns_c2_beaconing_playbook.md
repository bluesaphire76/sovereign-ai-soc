# DNS Command-and-Control Beaconing Playbook

## Purpose

This playbook supports investigation of DNS patterns that may indicate command-and-control beaconing.

It focuses on repeated queries, regular intervals, unusual domains, host correlation, Suricata correlation, and safe containment decisions.

## When to Use

- A host repeatedly queries the same suspicious domain.
- DNS queries occur at stable or near-stable intervals.
- A domain is newly observed, rare, high-risk, or associated with suspicious infrastructure.
- DNS activity aligns with Suricata alerts or host compromise indicators.
- The querying host is sensitive, internet-facing, or recently involved in authentication anomalies.

## Detection Signals

- Repeated DNS queries from one host to one domain or parent domain.
- Query intervals that look automated rather than user-driven.
- Suspicious TLD, dynamic DNS, newly registered domain, or poor reputation domain.
- DNS lookups followed by outbound connections to resolved IPs.
- Suricata alert involving the same host, domain, SNI, or destination IP.
- DNS activity after suspicious login, package install, new service, or process execution.

## Initial Triage

- Identify requesting host, user context, domain, query type, resolver, first seen, last seen, and query count.
- Determine whether the domain belongs to approved software, update service, monitoring platform, or business application.
- Review query timing to identify regular intervals.
- Check whether DNS responses resolved to external IPs and whether the host connected to them.
- Determine whether other hosts queried the same domain.
- Review whether the host has recent Wazuh or Suricata alerts.

## Evidence to Collect

- DNS query records with timestamps and query types.
- Domain reputation, registration age, ASN, resolved IPs, and passive DNS context where available.
- Network flow or proxy evidence to resolved IPs.
- Suricata alerts linked to the domain, SNI, destination IP, or host.
- Host telemetry for suspicious login, sudo, process execution, package activity, and service changes.
- Business owner validation for the domain or application.
- Historical query baseline for the host and domain.

## Investigation Steps

1. Build a DNS timeline for the host and domain.
2. Calculate whether query intervals are periodic, clustered, or user-driven.
3. Check whether the parent domain is newly observed in the environment.
4. Validate whether the domain is known SaaS, CDN, update, monitoring, EDR, or internal service.
5. Review whether the host made outbound connections after DNS resolution.
6. Correlate with Suricata alerts for the same destination, SNI, or flow.
7. Review host events around the first query time for suspicious login, package install, service creation, or process execution.
8. Determine whether the activity is expected software telemetry, suspicious beaconing, or likely C2.

## Correlation Checks

- Correlate with Suricata high severity alerts.
- Correlate with suspicious systemd services or package activity.
- Correlate with authentication anomalies before first DNS query.
- Correlate with proxy, firewall, and outbound flow logs.
- Correlate with other hosts querying the same domain.
- Correlate with threat intelligence when available.

## False Positive Conditions

- Domain belongs to approved update service, EDR, telemetry agent, monitoring tool, or SaaS platform.
- Query interval matches expected software heartbeat.
- Business owner confirms the application and domain.
- Many managed hosts query the same vendor domain as part of standard operation.
- No resolved-IP connections occur, or connections are confirmed benign.
- Domain is internal or controlled by the organization.

## Escalation Criteria

- Domain is newly registered, rare, suspicious, or linked to threat intelligence.
- Query interval is highly regular and not tied to known software.
- Host also shows suspicious login, sudo, package, service, or process activity.
- Suricata flags the same destination or protocol.
- Multiple internal hosts query the same suspicious domain.
- Host communicates with resolved IPs on unusual ports.

## Containment Actions

- Preserve DNS, flow, Suricata, and host evidence before blocking.
- Block the domain or resolved IP after validation and approval.
- Isolate the host if DNS beaconing correlates with endpoint compromise.
- Temporarily increase monitoring for the host and domain.
- Avoid enterprise-wide blocking before checking business ownership.

## Remediation Actions

- Remove malware, persistence, or unauthorized software if host compromise is confirmed.
- Rotate credentials if suspicious authentication preceded beaconing.
- Review host startup services, scheduled jobs, and package changes.
- Add domain or destination detection when malicious infrastructure is confirmed.
- Document allowed domain if the behavior is benign and recurring.

## Closure Criteria

- Domain ownership and business purpose are validated or malicious classification is documented.
- DNS interval, query volume, and resolved-IP connections are reviewed.
- Host and Suricata correlations were checked.
- Containment or monitoring decision is recorded.
- False-positive rationale includes owner or vendor evidence.
- Residual risk is documented when the domain is not blocked.

## Analyst Notes

- Periodic DNS alone does not prove C2; many enterprise tools beacon.
- C2 suspicion increases when DNS behavior aligns with host compromise signals.
- Domain reputation should support, not replace, local evidence.
- Treat newly seen domains from sensitive hosts as higher risk.
