# v0.3 PostgreSQL 18.4 Docker Upgrade

## Scope

Upgrade local AI SOC PostgreSQL runtime from PostgreSQL 16 to PostgreSQL 18.4 using dump/restore and a new Docker volume.

## Runtime state after upgrade

- Container name: `postgres-soc`
- Image: `postgres:18.4`
- Volume: `postgres_soc_data_pg18`
- Bind address: `127.0.0.1:5432`
- Database: `aisoc`
- User: `soc`

## Previous rollback state

The previous PostgreSQL 16 runtime was preserved as:

- Container: `postgres-soc-pg16-rollback`
- Volume: `postgres_soc_data`

## Important PostgreSQL 18+ Docker note

For PostgreSQL 18+ Docker images, the volume must be mounted at:

```bash
/var/lib/postgresql
```

not at:

```bash
/var/lib/postgresql/data
```

Using `/var/lib/postgresql/data` with `postgres:18.4` can cause the container to restart with a data directory layout error.

## Migration method

1. Stop AI SOC frontend/backend/worker.
2. Create a custom-format dump from PostgreSQL 16 using `pg_dump -Fc`.
3. Preserve the old PostgreSQL 16 container and volume for rollback.
4. Start PostgreSQL 18.4 on a new Docker volume.
5. Restore the dump using `pg_restore --no-owner --no-acl`.
6. Validate DB connectivity through the application Python runtime.
7. Restart AI SOC services.
8. Validate Wazuh ingest, worker heartbeat and incident creation.

## Post-upgrade ingest recovery

After the upgrade, Wazuh Indexer was receiving current alerts and the worker was running, but the worker was repeatedly reading the same duplicate alert batch.

Root cause:

- `wazuh_ingest_watermarks.last_timestamp` was stale.
- The worker query uses `last_timestamp - overlap`.
- With a dense overlap window and `WAZUH_BATCH_SIZE=50`, the worker could keep seeing the same 50 duplicate documents and never advance to newer alerts.

Recovery applied:

1. Stop `ai-soc-worker`.
2. Backup `wazuh_ingest_watermarks`.
3. Reset the Wazuh ingest watermark to current time minus a few minutes.
4. Restart `ai-soc-worker`.
5. Validate new events in `incidents`.

## Follow-up hardening recommendation

Before closing the PostgreSQL 18.4 upgrade point, apply a small worker hardening patch to reduce the risk of the ingest worker getting stuck on a duplicate-only batch again.

Recommended actions:

- Increase `WAZUH_BATCH_SIZE`, for example to `500`.
- Reduce `WAZUH_WATERMARK_OVERLAP_SECONDS`, for example to `15`.
- Improve worker logic so a full duplicate batch can still advance the watermark safely.
- Keep PostgreSQL 18.4 mounted at `/var/lib/postgresql`, not `/var/lib/postgresql/data`.

## Validation checklist

The upgrade can be considered validated when:

- `postgres-soc` is running with image `postgres:18.4`.
- PostgreSQL reports server version `18.4`.
- API and frontend services are active.
- AI SOC API health returns `status: ok`.
- Wazuh Indexer receives recent alerts.
- `ai-soc-worker` heartbeat is recent.
- `wazuh_ingest_watermarks.last_timestamp` advances.
- `incidents.timestamp` receives new events after the migration.
- No continuous duplicate-only loop is visible in `ai-soc-worker` logs.
