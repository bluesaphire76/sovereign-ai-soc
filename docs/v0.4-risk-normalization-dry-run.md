# v0.4 Risk Normalization Dry Run

## Purpose

This dry-run validates the impact of the v0.4 risk/severity normalization policy on historical correlated incidents without modifying production data.

## Summary

The previous scoring model over-classified many operational or repetitive events as `CRITICAL` / `ESCALATED` because correlation scoring accumulated base score, pattern score, volume score and chain bonus without a strict cap for non-attack-chain events.

The v0.4 normalization policy introduces caps based on Wazuh level and requires strong correlation or critical attack-chain evidence before automatic `CRITICAL` / `ESCALATED` classification.

## Dry-run result

See:

```text
reports/validation/v0.4-risk-normalization-dry-run.txt
```

Key findings:

- 4,750 correlated incidents evaluated.
- 3,576 incidents would change under the new policy.
- `CRITICAL` priority would significantly decrease.
- `ESCALATED` status would significantly decrease.
- Many operational events such as PAM sessions, sudo operational activity, AppArmor messages, Systemd service failures and dpkg changes would no longer be treated as `CRITICAL` without attack-chain evidence.

## Decision

No historical production data was modified in this step.

Historical backfill should be handled separately with:

- explicit dry-run mode by default
- explicit `--apply` flag
- database backup before execution
- audit/export file
- exclusion of `CLOSED` and `FALSE_POSITIVE` records
- limited scope for operational noise and non-attack-chain incidents
