# v0.6 Step 20 - Controlled SOAR Remediation Actions

Step 20 introduces the first controlled SOAR remediation action workflow.

This is not arbitrary command execution and it is not autonomous remediation. The implementation supports only safe, allowlisted product workflow actions such as creating an internal remediation task, incident note, case action or audit entry.

## Execution Boundary

The execution path is intentionally constrained:

```text
LLM recommendation
 -> normalized remediation action type
 -> controlled SOAR allowlist
 -> human approval confirmation
 -> AI governance check
 -> dry-run check
 -> rollback readiness check
 -> internal product workflow record
 -> audit before and after result
```

Forbidden path:

```text
LLM text -> shell command -> execution
```

The system does not execute shell commands, SSH commands, firewall changes, identity changes, host isolation, process termination, file quarantine or Wazuh active response in this step.

## LLM Model Selection

Remediation intelligence continues to use the shared LLM task policy:

- `task=REMEDIATION`
- `user_triggered=True`
- incident severity / recommended priority is passed into policy selection

With the current three configured profiles, remediation uses the `quality` profile for user-triggered HIGH/CRITICAL incidents and `standard` otherwise. The active model metadata is exposed in the remediation response as:

- `model_profile`
- `model`
- `model_task`

The LLM remains advisory only. It can propose structured remediation recommendations, but it never generates executable commands and its text is never dispatched directly.

## Supported MVP Actions

The controlled SOAR executor maps only low-risk remediation action types to internal product workflow templates:

| Remediation action type | Controlled SOAR template |
|---|---|
| `CREATE_TICKET` | `CREATE_REMEDIATION_TASK` |
| `NOTIFY_OWNER` | `NOTIFY_OWNER_MANUAL_TASK` |
| `ESCALATE_CASE` | `CREATE_CASE_ACTION` |
| `COLLECT_FORENSIC_EVIDENCE` | `GENERATE_OPERATOR_CHECKLIST` |

If the incident is linked to a case, the workflow creates a `CaseAction` and `CaseAudit`. If no linked case exists, it creates an `IncidentNote` and incident audit records.

Every request records an incident audit event before gate evaluation and another after the result. Security audit metadata is also recorded for governance traceability.

## Unsupported Actions

High-impact or external actions return `NOT_SUPPORTED`:

- `DISABLE_USER`
- `ISOLATE_HOST`
- `BLOCK_IP`
- `KILL_PROCESS`
- `QUARANTINE_FILE`
- `STOP_SERVICE`
- `RESTART_SERVICE`
- `ADD_FIREWALL_RULE`

These require future external connectors, whitelisted playbooks and stronger execution governance.

## API

```http
POST /incidents/{incident_id}/remediation-actions/{action_id}/execute-approved
```

RBAC:

- `ADMIN`: allowed
- `ANALYST`: allowed
- `VIEWER`: denied by RBAC

Request body:

```json
{
  "approval_confirmed": true,
  "approval_rationale": "Operator confirmed controlled product-only workflow action from Incident Command Room."
}
```

The body does not accept free-form command payloads or target instructions.

## Frontend

Incident Detail / Command Room now shows controlled SOAR support in the Replay Simulation action list.

Supported actions show:

- support status
- controlled template
- policy gate state
- human approval confirmation checkbox
- `Execute approved action` button

The button is disabled for:

- VIEWER users
- unsupported actions
- blocked dry-run
- blocked rollback readiness
- blocked governance
- missing confirmation

The UI clearly states that only internal task, note or audit records are created.

## Non-Goals

Step 20 does not implement:

- arbitrary shell execution
- unrestricted SSH
- real endpoint action
- firewall modification
- user disablement
- service stop/restart
- process kill
- file quarantine
- host isolation
- autonomous remediation

The value of this step is governed product workflow execution, not production-impacting remediation.
