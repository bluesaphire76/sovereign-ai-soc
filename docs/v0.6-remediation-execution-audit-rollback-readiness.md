# v0.6 Step 10 - Remediation Execution Audit and Rollback Readiness Foundation

## Purpose

Step 10 adds governance structures that must exist before any future safe remediation executor can be considered.

Step 10 does not execute remediation actions.

Execution readiness is a governance assessment, not an execution capability.

The implementation models:

- execution readiness assessment
- rollback readiness assessment
- execution audit record preparation
- chain of custody between investigation evidence, remediation plan, approval, dry-run and readiness
- audit-friendly event objects that do not imply real execution

No endpoint, frontend control, persistence migration or executor is added in this step.

## Architecture

The implementation extends the existing `remediation` package:

- `rollback_readiness.py` assesses action-level and plan-level rollback readiness.
- `readiness.py` assesses whether an action is eligible for a future executor review.
- `execution_audit.py` prepares execution audit records and chain-of-custody objects.
- `audit.py` adds sanitized readiness, rollback and chain-of-custody audit events.
- `validators.py` validates readiness, rollback readiness and execution audit records.

The modules remain domain-only and deterministic. They do not call a shell, contact target hosts, modify Wazuh or Suricata, or write production configuration.

`api.py` is not extended with business logic.

## No-Execution Boundary

The Step 10 boundary is strict:

- no command execution
- no SSH execution
- no Wazuh active response execution
- no firewall changes
- no user changes
- no service stop or restart
- no file quarantine
- no process kill
- no production configuration writes
- no execute endpoint

The execution audit record explicitly tracks:

- `execution_supported = false`
- `execution_attempted = false`

Readiness may return `READY_FOR_FUTURE_EXECUTOR`, but this means the governance data is complete enough for a future executor design review. It does not mean execution is available.

## Execution Readiness Model

Readiness checks are deterministic and conservative.

The readiness service checks:

1. Plan/action context exists.
2. The action is not forbidden.
3. Approval exists where required.
4. Approval has no policy issues.
5. Dry-run output is available or generated from metadata.
6. Rollback readiness is acceptable.
7. Evidence references exist.
8. Action type is whitelisted for future executor compatibility.
9. Command preview is non-executable and does not contain shell-like operators.
10. No execution flag is set.

Supported execution audit statuses:

- `NOT_SUPPORTED`
- `READY_FOR_FUTURE_EXECUTOR`
- `BLOCKED_BY_POLICY`
- `BLOCKED_BY_MISSING_APPROVAL`
- `BLOCKED_BY_MISSING_ROLLBACK`
- `BLOCKED_BY_VALIDATION`
- `EXECUTION_NOT_IMPLEMENTED`

The model intentionally avoids `SUCCESS` and `FAILED` execution outcomes because no execution occurs in Step 10.

## Rollback Readiness Model

Rollback readiness checks:

- rollback steps
- rollback validation notes
- rollback feasibility
- limitations
- blockers
- recovery notes
- approval requirement
- risk context

Supported rollback statuses:

- `READY`
- `PARTIAL`
- `MISSING`
- `NOT_APPLICABLE`
- `BLOCKED`
- `UNKNOWN`

Missing or blocked rollback readiness prevents future execution readiness for operational actions.

## Chain of Custody

The chain-of-custody model links:

```text
investigation evidence
 -> remediation plan
 -> remediation action
 -> approval decision
 -> dry-run result
 -> rollback readiness
 -> execution readiness
 -> execution audit record
```

The chain records missing links as gaps so future persistence or UI layers can show exactly why an action is not ready for later executor review.

## Audit Events

Audit event objects use readiness-oriented wording:

- `REMEDIATION_EXECUTION_READINESS_ASSESSED`
- `REMEDIATION_EXECUTION_BLOCKED_BY_POLICY`
- `REMEDIATION_EXECUTION_BLOCKED_BY_MISSING_APPROVAL`
- `REMEDIATION_EXECUTION_BLOCKED_BY_MISSING_ROLLBACK`
- `REMEDIATION_ROLLBACK_READINESS_ASSESSED`
- `REMEDIATION_CHAIN_OF_CUSTODY_PREPARED`
- `REMEDIATION_UNSUPPORTED_EXECUTION_REQUESTED`

No audit event is named `remediation_executed`.

Audit details do not include executable command previews.

## Persistence

Step 10 does not add database tables. Persistence is deferred intentionally to avoid coupling the governance model to an execution workflow that does not exist yet.

Future persistence candidates:

- `remediation_execution_audit`
- `remediation_readiness_assessments`
- `remediation_rollback_readiness`

Those tables should be added only when the API and UI review workflows are ready.

## Future Executor Prerequisites

Before any future executor is considered, the product must add:

- explicit executor allowlist
- endpoint and RBAC design
- persisted approval records
- persisted readiness records
- rollback execution model
- execution evidence capture
- executor-specific audit trail
- controlled failure handling
- clear UI that distinguishes readiness from execution

## Limitations

Step 10 is not a SOAR engine. It does not perform active response. It does not modify host, network, identity, Wazuh, Suricata or firewall state.

The value of this step is governance: it prepares an audit-ready, rollback-aware foundation while preserving the non-autonomous remediation boundary.
