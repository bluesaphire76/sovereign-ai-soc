# v0.6 Investigation Memory and Persistence Foundation

## Purpose

Step 5 introduces structured investigation persistence for Sovereign AI SOC.

This is structured investigation persistence, not autonomous AI memory.

Raw chain-of-thought style reasoning must not be persisted.

The objective is to make investigation output replayable, auditable and versioned while keeping the platform analyst-centric and governed.

## Architecture

The Step 5 implementation is centered on `investigation_ai/persistence.py`.

The persistence layer is separate from routers and `api.py`. It stores structured investigation artifacts generated by the investigation engine:

- investigation sessions
- investigation snapshots
- hypothesis history
- confidence history
- retrieval enrichment metadata
- analyst feedback foundation

The investigation engine can optionally persist generated briefs when a database session is provided. If persistence fails, the generated brief remains valid and rendering should continue.

## Database Structures

Step 5 adds SQLAlchemy models for:

- `investigation_sessions`
- `investigation_snapshots`
- `investigation_hypothesis_history`
- `investigation_confidence_history`
- `investigation_retrieval_history`
- `investigation_feedback`

The schema can be ensured with:

```bash
python3 scripts/create_investigation_memory_tables.py
```

The script follows the existing repository convention of loading `Base.metadata` and creating missing tables.

## Session Lifecycle

An investigation session captures the current state of an investigation for one incident.

Tracked fields include:

- session id
- incident id
- status
- generated by
- model name
- parent session id
- investigation version
- enrichment pass count
- fallback usage
- confidence score and level
- created and updated timestamps

The session record represents the latest known state. Snapshots retain versioned historical state.

## Snapshot Versioning

Every persisted brief creates a new snapshot:

```text
Investigation Snapshot v1
Investigation Snapshot v2
Investigation Snapshot v3
```

Snapshots store the serialized structured `InvestigationBrief`, not uncontrolled prompt text.

Each snapshot tracks:

- version
- snapshot type
- evidence count
- hypothesis count
- recommended check count
- recommended action count

This creates the foundation for future investigation replay.

## Confidence Evolution

Confidence changes are tracked in `investigation_confidence_history`.

The record captures:

- previous score and level
- new score and level
- investigation version
- snapshot id
- change reason

This allows future views to explain how confidence evolved after enrichment, contradiction handling or analyst feedback.

## Hypothesis Evolution

Each persisted brief writes hypothesis history rows.

Tracked values include:

- hypothesis id
- hypothesis status
- confidence score
- claim classification
- supporting evidence count
- contradictory evidence count
- missing evidence count

This supports future replay of hypothesis lifecycle states such as `ACTIVE`, `CONFIRMED`, `WEAKENED`, `DISCARDED` and `NEEDS_EVIDENCE`.

## Retrieval Metadata

Step 4 retrieval enrichment metadata can be persisted with a snapshot.

Stored metadata includes:

- retrieval request id
- retrieval type
- retrieval status
- duration
- evidence returned
- limits applied
- failures
- audit summary

This supports governance and replay without creating autonomous retrieval loops.

## Analyst Feedback Foundation

The `investigation_feedback` table provides a safe foundation for analyst feedback.

It can store:

- analyst
- feedback type
- feedback text
- confidence override
- hypothesis reference

This step does not implement analyst coaching or self-learning behavior.

## Persistence Boundaries

The persistence layer must not store:

- raw unrestricted prompts
- arbitrary chain-of-thought
- hidden AI scratchpads
- uncontrolled AI memory blobs
- remediation execution state

Only structured investigation artifacts are persisted.

## Engine Integration

`generate_investigation_brief(...)` accepts an optional persistence session.

When provided, the engine persists:

- deterministic briefs
- enriched deterministic briefs
- LLM-backed structured briefs
- deterministic fallback briefs

Persistence is best-effort. Failures are logged and converted into safe fallback behavior so incident rendering is not blocked.

## Limitations

Step 5 does not add:

- new frontend UI
- new API endpoints
- full replay UI
- cross-incident intelligence
- autonomous long-term AI memory
- remediation execution
- active response
- SOAR orchestration

The work is a persistence and governance foundation for future replayable enterprise investigations.
