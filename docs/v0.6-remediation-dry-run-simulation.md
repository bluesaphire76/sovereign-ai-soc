# v0.6 Step 14 - Remediation Dry-Run Simulation

Step 14 adds a read-only remediation dry-run simulation layer for the AI remediation workflow.

The feature connects the existing remediation intelligence plan to the governed dry-run, rollback readiness, approval gate and execution readiness models. It is intended to help an analyst understand what would block or allow a future governed remediation review.

## Boundary

This step does not execute remediation actions.

The dry-run endpoint and UI do not:

- run shell commands
- contact target hosts
- change firewall rules
- disable users
- quarantine files
- stop or restart services
- mutate production state
- dispatch remediation actions

The response always declares:

- `execution_supported: false`
- `state_mutated: false`
- `human_approval_required: true`

## Backend Flow

The read-only endpoint is:

```text
GET /incidents/{incident_id}/remediation-dry-run
```

The service flow is:

```text
incident remediation intelligence
 -> dry-run action adaptation
 -> approval gate evaluation
 -> rollback readiness assessment
 -> execution readiness assessment
 -> structured simulation response
```

The router delegates to `remediation.simulation` and contains no remediation business logic. `api.py` only contains the RBAC route entry.

## Response Semantics

The dry-run status can be:

- `READY_FOR_REVIEW`
- `MISSING_APPROVAL`
- `MISSING_EVIDENCE`
- `MISSING_ROLLBACK`
- `BLOCKED_BY_POLICY`
- `NOT_SUPPORTED`

Findings explain approval gaps, rollback gaps, missing evidence, policy blockers and the no-execution boundary.

## Frontend

Incident Detail includes a compact **Dry-run simulation** section in the investigation console.

The dry-run is loaded asynchronously outside the main incident `Promise.all` flow and displays:

- simulation status
- execution support and state mutation flags
- approval requirement
- top findings
- rollback readiness
- blockers
- next safe steps

The UI exposes no execution controls.

## Governance

The dry-run is a review artifact only. It prepares later governance and execution-readiness workflows but does not approve or execute remediation.
