# v0.7 Qdrant Semantic Memory Foundation

AI SOC uses Qdrant as a local semantic memory layer for SOC knowledge and investigation support.

This capability is part of the v0.7.0 roadmap and is introduced as **Step 15 — Qdrant Semantic Memory**.

---

## Purpose

Qdrant is used as a **semantic memory and retrieval layer** for AI SOC.

Allowed use cases:

- semantic search on historical/security context;
- retrieval of playbooks and operating procedures;
- SOC documentation RAG;
- AI Analysis context enrichment;
- Detection Quality and Detection Control Plane decision support.
- governed Detection Control and Case Closure memory review.

The objective is to help analysts answer questions such as:

```text
Has something similar happened before?
Which playbook is most relevant for this incident?
Which SOC procedure applies here?
What historical context could enrich the AI analysis?
Are there similar exceptions or suppressions already configured?
```

---

## Decision Boundary

Qdrant is a **support system only**.

It must **not** be used for:

- primary operational deduplication;
- final severity decisions;
- automatic noise suppression;
- automatic incident closure;
- final classification without deterministic verification;
- replacing correlation rules.

Deterministic controls, RBAC, audit, validation and human review remain authoritative.

The guiding principle is:

```text
Qdrant can retrieve context.
Qdrant must not make operational decisions.
```

---

## Allowed Uses

### 1. Semantic search on historical incidents

Example question:

```text
Does this new incident resemble something already seen before?
```

Qdrant can help retrieve similar historical context, but the result must be presented as **analyst support**, not as an automatic classification.

---

### 2. Playbook and procedure retrieval

Example question:

```text
Which playbook should be used for a suspicious DNS-related Suricata alert?
```

Qdrant can retrieve relevant playbooks from the local SOC knowledge base.

---

### 3. SOC documentation RAG

Example question:

```text
How should I handle a Suricata incident with suspicious DNS but no endpoint compromise?
```

Qdrant can retrieve local documentation snippets that enrich the answer.

---

### 4. AI Analysis and AI Brief enrichment

Qdrant may provide additional context to Incident AI Brief and Case AI Analysis, such as:

- similar historical incidents;
- relevant operating procedures;
- related playbooks;
- previous analyst guidance;
- known false-positive patterns.

This context must be clearly marked as **Retrieved Semantic Memory Context** and
as **advisory only**, not deterministic evidence.

Prompt-visible semantic memory blocks must include:

- source;
- semantic score;
- chunk index;
- text excerpt;
- usage rules forbidding final severity, operational deduplication, automatic
  noise suppression, incident/case closure and deterministic correlation
  replacement.

---

### 5. Detection Quality and tuning support

Qdrant may support Detection Quality and Detection Control Plane workflows by retrieving:

- similar detection rules;
- similar exceptions;
- similar noise suppression cases;
- previous tuning rationale;
- related documentation.

However, Qdrant must remain **decision support only**.

---

## Forbidden Uses

Qdrant must not be used for the following:

```text
primary operational deduplication
final severity decision
automatic noise suppression
automatic incident closure
final incident classification without deterministic verification
replacement of correlation rules
```

These decisions must remain governed by deterministic logic, policy, RBAC, audit and analyst review.

## Read-only API Foundation

Step 15A introduces read-only semantic memory endpoints.

```text
GET /semantic-memory/capabilities
GET /semantic-memory/health
GET /semantic-memory/collection
GET /semantic-memory/index-status
GET /semantic-memory/search?query=<text>&limit=4
POST /semantic-memory/historical-backfill
POST /semantic-memory/detection-case-backfill
POST /semantic-memory/retention-cleanup
GET /incidents/{incident_id}/similar-incidents
POST /detection-quality/semantic-context
```

These endpoints are intended to make Qdrant:

- visible;
- testable;
- observable;
- governed;
- ready for later controlled integrations.

They do **not** modify incidents, cases, detection rules, severity, status, suppression, deduplication or correlation state.

---

## Endpoint: `/incidents/{incident_id}/similar-incidents`

Purpose:

```text
Retrieve historical incident semantic memory similar to one selected incident.
```

Expected information:

- selected incident id;
- result count;
- historical incident id;
- semantic score;
- historical status and risk metadata;
- source;
- short excerpt;
- advisory-only decision boundary.

Important:

```text
Similar incidents are historical context only.
Similarity does not prove same root cause.
Similarity does not mean duplicate.
Similarity must not determine severity.
Similarity must not close or suppress an incident.
Human validation remains required.
```

Access is initially limited to `ADMIN` and `ANALYST`.

---

## Endpoint: `/detection-quality/semantic-context`

Purpose:

```text
Retrieve semantic memory context for Detection Quality review.
```

Expected information:

- related playbook or procedure snippets;
- possible historical false-positive or noisy-pattern context;
- possible historical tuning examples;
- result count;
- advisory-only decision boundary.

Important:

```text
Semantic memory context is advisory only.
It must not create, update, approve or apply detection rules, exceptions or noise suppression.
It must not determine final severity, deduplication, incident closure or tuning decisions.
Human validation and governed Detection Control approval remain required.
```

Access is initially limited to `ADMIN` and `ANALYST`.

## Re-indexing Runbook

Qdrant indexing is an operator-controlled CLI action.

Use this when `knowledge_base/security_playbook.md` or other configured
knowledge base documents change. Running without `--recreate` preserves
existing historical incident memory points:

```bash
PYTHONPATH=. .venv/bin/python rag_index.py
```

Use recreate only when the whole collection must be rebuilt. Recreate removes
the collection first, so historical incident memory must be re-applied after it:

```bash
PYTHONPATH=. .venv/bin/python rag_index.py --recreate
PYTHONPATH=. .venv/bin/python scripts/index_historical_incidents_to_qdrant.py --apply --limit 10000
```

Historical incident memory indexing is also operator-controlled:

```bash
PYTHONPATH=. .venv/bin/python scripts/index_historical_incidents_to_qdrant.py --dry-run
PYTHONPATH=. .venv/bin/python scripts/index_historical_incidents_to_qdrant.py --apply
```

Detection Control and Case Closure memory indexing is operator-controlled:

```bash
PYTHONPATH=. .venv/bin/python scripts/index_detection_case_memory_to_qdrant.py --dry-run
PYTHONPATH=. .venv/bin/python scripts/index_detection_case_memory_to_qdrant.py --apply
```

Case Closure memory includes only final/approved closure outcomes:

- checklist explicitly approved by a human reviewer; or
- linked case already in a final status such as `CLOSED`, `RESOLVED` or
  `FALSE_POSITIVE`.

Draft closure checklists from open, triaged or investigating cases are skipped
so semantic memory does not learn unfinished case decisions. Apply mode also
prunes stale `detection_control` and `case_closure` points after a complete scan,
without touching `knowledge_base` or `historical_incident` memory.

Historical incident memory cleanup is dry-run by default:

```bash
PYTHONPATH=. .venv/bin/python scripts/qdrant_memory_retention.py --dry-run
```

Apply cleanup only after reviewing dry-run output:

```bash
PYTHONPATH=. .venv/bin/python scripts/qdrant_memory_retention.py --apply
```

The cleanup script targets only `source_type=historical_incident` and never
deletes `source_type=knowledge_base`.

The Semantic Memory settings page exposes admin-only operational buttons for:

- historical backfill dry-run;
- historical backfill apply;
- Detection Control and Case Closure backfill dry-run;
- Detection Control and Case Closure backfill apply;
- retention cleanup dry-run;
- retention cleanup apply.

Apply actions require explicit confirmation and are blocked for non-admin roles.

Optional systemd timers are provided for periodic operations:

```text
deploy/systemd/ai-soc-qdrant-historical-backfill.service
deploy/systemd/ai-soc-qdrant-historical-backfill.timer
deploy/systemd/ai-soc-qdrant-detection-case-backfill.service
deploy/systemd/ai-soc-qdrant-detection-case-backfill.timer
deploy/systemd/ai-soc-qdrant-retention-dry-run.service
deploy/systemd/ai-soc-qdrant-retention-dry-run.timer
```

Install or refresh them with:

```bash
sudo cp deploy/systemd/ai-soc-qdrant-historical-backfill.service /etc/systemd/system/
sudo cp deploy/systemd/ai-soc-qdrant-historical-backfill.timer /etc/systemd/system/
sudo cp deploy/systemd/ai-soc-qdrant-detection-case-backfill.service /etc/systemd/system/
sudo cp deploy/systemd/ai-soc-qdrant-detection-case-backfill.timer /etc/systemd/system/
sudo cp deploy/systemd/ai-soc-qdrant-retention-dry-run.service /etc/systemd/system/
sudo cp deploy/systemd/ai-soc-qdrant-retention-dry-run.timer /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now ai-soc-qdrant-historical-backfill.timer
sudo systemctl enable --now ai-soc-qdrant-detection-case-backfill.timer
sudo systemctl enable --now ai-soc-qdrant-retention-dry-run.timer
```

The scheduled backfills are idempotent and index only eligible historical,
Detection Control and final/approved Case Closure memory. Detection/case
backfill apply also removes stale detection/case points after a complete scan.
The scheduled retention unit is dry-run only; destructive cleanup remains a
reviewed admin action from CLI or GUI.

Then verify the index without modifying it:

```bash
curl -s http://127.0.0.1:8008/semantic-memory/index-status \
  -H "Authorization: Bearer $TOKEN" \
| jq '.status, .points_count, .documents_count, .documents[0].source, .indexing_mode'
```

Expected:

```text
status is OK
points_count is greater than zero
source includes knowledge_base/security_playbook.md
indexing_mode is manual_cli_only
```
