# Sovereign AI SOC v0.6.0 — Observability Architecture and Operations Guide

## Purpose

This document describes the observability architecture introduced in the v0.6.0 evolution of Sovereign AI SOC.

The goal is to provide:

- an enterprise-native Health page inside the application;
- historical metrics through Prometheus;
- operational dashboards through Grafana;
- custom AI SOC metrics for ingestion, worker health, AI runtime, triage outcomes and active users;
- infrastructure visibility including host, runtime/cgroup and GPU metrics;
- troubleshooting guidance for local-first SOC operations.

The Health page and Grafana serve different purposes and intentionally remain separate.

---

## Architecture decision

### Health page

The in-application Health page remains the primary enterprise-native operational view.

It is intended for authenticated SOC users and provides application-aware status such as:

- platform posture;
- component health;
- worker status;
- ingestion freshness;
- AI runtime status;
- latest incident context;
- active users.

Grafana does not replace the Health page.

### Grafana

Grafana runs as a separate Docker-based observability layer.

It provides:

- historical trends;
- infrastructure visibility;
- GPU visibility;
- Prometheus target health;
- SOC/AI pipeline metrics;
- operational troubleshooting dashboards.

Grafana is not embedded in the application through iframe. The application exposes only a lightweight external Observability link.

### Prometheus

Prometheus stores historical metrics and scrapes:

- Prometheus itself;
- node-exporter;
- cAdvisor;
- AI SOC API `/metrics`.

## Security model

Current design:

- Grafana is local-only on `127.0.0.1:3002`.
- Prometheus is local-only on `127.0.0.1:9090`.
- Exporters are local-only where possible.
- `/metrics` is unauthenticated for Prometheus scraping but must not be exposed publicly.
- The application Observability link is visible to `ADMIN` and `ANALYST`.
- The Observability link is not visible to `VIEWER`.
- No iframe embedding is used.

If Grafana is later exposed through Nginx or Cloudflare Access, Prometheus and exporter endpoints should remain private.

## Runtime / cgroup metrics

In WSL, cAdvisor may expose runtime/cgroup data through the `id` label rather than Docker container names.

Example:

```text
id="/system.slice/ai-soc-api.service"
id="/system.slice/ai-soc-worker.service"
```

The Foundation dashboard uses `id`-based queries.

Example PromQL:

```promql
topk(10, sum by (id) (rate(container_cpu_usage_seconds_total{cpu="total",id!~"/|/system.slice|/init.scope"}[5m])))
```

```promql
topk(10, container_memory_usage_bytes{id!~"/|/system.slice|/init.scope"})
```

## AI triage timeout hardening

v0.6.0 introduces safer local AI timeout handling.

Configuration:

```text
AI_TRIAGE_ENABLED=true
AI_TRIAGE_TIMEOUT_SECONDS=30
AI_TRIAGE_RETRY_ON_INVALID_OUTPUT=true
AI_TRIAGE_RETRY_ON_TIMEOUT=true
AI_TRIAGE_RETRY_TIMEOUT_SECONDS=15
AI_TRIAGE_COMPACT_RETRY_MAX_CHARS=3000
AI_TRIAGE_FALLBACK_ON_ERROR=true
```

Behavior:

1. The first local AI triage call uses `AI_TRIAGE_TIMEOUT_SECONDS`.
2. If the call times out, one compact retry is attempted.
3. The compact retry uses reduced prompt context.
4. If retry fails, deterministic fallback is used.
5. The fallback reason is explicit and analyst-readable.

Example fallback text:

```text
Fallback reason: Local AI triage timed out after 30s; compact retry also timed out after 15s.
Retry attempted: yes
Context note: Compact timeout retry was attempted with reduced prompt context to protect worker throughput.
```

This preserves safe deterministic fallback while improving resilience and analyst clarity.


## Non-goals

The v0.6.0 observability layer intentionally does not:

- replace the Health page;
- embed Grafana through iframe;
- expose Prometheus publicly;
- use Grafana as an authorization layer for SOC application users;
- perform automatic remediation;
- bypass human-in-the-loop controls;
- change investigation or remediation database schemas as part of observability.

---

## Summary

The v0.6.0 observability work turns Sovereign AI SOC into a more measurable SOC platform.

It adds:

- local-first observability;
- historical metrics;
- infrastructure and GPU visibility;
- SOC ingestion quality metrics;
- AI runtime status;
- AI triage behavior visibility;
- noise reduction visibility;
- active user visibility;
- operational troubleshooting guidance.

The Health page remains the enterprise-native in-app view. Grafana and Prometheus provide the historical observability layer.
