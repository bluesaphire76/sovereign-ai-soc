# v0.7.0 Alertmanager Wazuh Backlog Alerting

## Purpose

This step adds **Prometheus Alertmanager** to the AI SOC observability stack to detect and route operational alerts related to:

- Wazuh ingest backlog
- Wazuh ingest watermark lag
- AI SOC worker heartbeat freshness
- AI SOC API metrics availability
- Wazuh ingest component error state

The goal is to move from passive observability to proactive alerting behavior when the platform risks accumulating unprocessed events or losing operational freshness.

---

## Architecture

```text
AI SOC API / Worker / Health metrics
        ↓
Prometheus scrape
        ↓
Prometheus alerting rules
        ↓
Alertmanager
        ↓
Notification receiver
        ↓
Operator action
```

Current implementation:

- Prometheus scrapes AI SOC metrics from `/metrics`.
- Prometheus evaluates alerting rules from `/etc/prometheus/rules/*.yml`.
- Alertmanager receives alerts from Prometheus on `127.0.0.1:9093`.
- Grafana remains the historical observability layer.
- AI SOC Health remains the native operational health view.
- The initial Alertmanager configuration intentionally uses null receivers until the Prometheus → Alertmanager loop is validated.

## Docker Compose Changes

Alertmanager is added to the existing observability compose stack.

Because Prometheus currently runs with `network_mode: host`, Alertmanager is also configured with `network_mode: host`.

Prometheus therefore routes alerts to:

```text
127.0.0.1:9093
```

and not to:

```text
alertmanager:9093
```

### Alertmanager service

```yaml
  alertmanager:
    image: prom/alertmanager:latest
    container_name: ai-soc-alertmanager
    restart: unless-stopped
    network_mode: host
    command:
      - "--config.file=/etc/alertmanager/alertmanager.yml"
      - "--storage.path=/alertmanager"
      - "--web.listen-address=127.0.0.1:9093"
    volumes:
      - ./alertmanager/alertmanager.yml:/etc/alertmanager/alertmanager.yml:ro
      - ai_soc_alertmanager_data:/alertmanager
```

### Prometheus rules volume

```yaml
    volumes:
      - ./prometheus/prometheus.yml:/etc/prometheus/prometheus.yml:ro
      - ./prometheus/rules:/etc/prometheus/rules:ro
      - ai_soc_prometheus_data:/prometheus
```

### Alertmanager volume

```yaml
volumes:
  ai_soc_prometheus_data:
  ai_soc_grafana_data:
  ai_soc_alertmanager_data:
```

## Prometheus Configuration

Prometheus must be configured to:

1. send alerts to Alertmanager;
2. load alerting rules from `/etc/prometheus/rules/*.yml`.

```yaml
alerting:
  alertmanagers:
    - static_configs:
        - targets:
            - 127.0.0.1:9093

rule_files:
  - /etc/prometheus/rules/*.yml
```

Expected beginning of `deploy/observability/prometheus/prometheus.yml`:

```yaml
global:
  scrape_interval: 15s
  evaluation_interval: 15s

alerting:
  alertmanagers:
    - static_configs:
        - targets:
            - 127.0.0.1:9093

rule_files:
  - /etc/prometheus/rules/*.yml

scrape_configs:
  - job_name: prometheus
```

## Alertmanager Configuration

File:

```text
deploy/observability/alertmanager/alertmanager.yml
```

Initial configuration:

```yaml
global:
  resolve_timeout: 5m

route:
  receiver: ai-soc-null
  group_by:
    - alertname
    - service
    - severity
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h

  routes:
    - matchers:
        - severity="critical"
      receiver: ai-soc-critical-null
      repeat_interval: 1h

receivers:
  - name: ai-soc-null

  - name: ai-soc-critical-null
```

This configuration is intentionally conservative.

It validates the Prometheus → Alertmanager delivery loop without introducing external notification secrets such as SMTP passwords, Telegram tokens, Slack webhooks, or Teams connectors.

## Alert Rules

File:

```text
deploy/observability/prometheus/rules/ai-soc-wazuh-backlog-alerts.yml
```

Rules:

```yaml
groups:
  - name: ai-soc-wazuh-ingest-backlog
    rules:
      - alert: WazuhIngestBacklogWarning
        expr: ai_soc_event_queue_pending_events > 250
        for: 15m
        labels:
          severity: warning
          service: ai-soc-worker
          domain: wazuh-ingest
        annotations:
          summary: "AI SOC Wazuh ingest backlog is growing"
          description: "Pending Wazuh events have been above 250 for more than 15 minutes. The worker may be falling behind."

      - alert: WazuhIngestBacklogCritical
        expr: ai_soc_event_queue_pending_events > 1000
        for: 10m
        labels:
          severity: critical
          service: ai-soc-worker
          domain: wazuh-ingest
        annotations:
          summary: "AI SOC Wazuh ingest backlog is critical"
          description: "Pending Wazuh events have been above 1000 for more than 10 minutes. Investigation data may be delayed."

      - alert: WazuhIngestWatermarkLagHigh
        expr: ai_soc_worker_batch_metrics{metric="watermark_lag_seconds"} > 1800
        for: 15m
        labels:
          severity: warning
          service: ai-soc-worker
          domain: wazuh-ingest
        annotations:
          summary: "AI SOC Wazuh ingest watermark lag is high"
          description: "The Wazuh ingest watermark has been more than 30 minutes behind for more than 15 minutes."

      - alert: AiSocWorkerHeartbeatStale
        expr: ai_soc_worker_heartbeat_age_seconds > 300
        for: 10m
        labels:
          severity: critical
          service: ai-soc-worker
          domain: worker-health
        annotations:
          summary: "AI SOC worker heartbeat is stale"
          description: "The latest AI SOC worker heartbeat has been older than 5 minutes for more than 10 minutes."

      - alert: AiSocApiMetricsDown
        expr: up{job="ai-soc-api"} == 0
        for: 5m
        labels:
          severity: critical
          service: ai-soc-api
          domain: observability
        annotations:
          summary: "AI SOC API metrics endpoint is down"
          description: "Prometheus cannot scrape the AI SOC API /metrics endpoint."

      - alert: WazuhIngestComponentError
        expr: ai_soc_component_status_code{component="wazuh_ingest",status="error"} == 1
        for: 5m
        labels:
          severity: critical
          service: ai-soc-worker
          domain: wazuh-ingest
        annotations:
          summary: "AI SOC Wazuh ingest component is reporting error"
          description: "The AI SOC platform health metrics report the Wazuh ingest component in error state."
```

---

## Metrics Used

| Metric | Purpose |
|---|---|
| `ai_soc_event_queue_pending_events` | Number of Wazuh events newer than the current ingest watermark |
| `ai_soc_worker_batch_metrics{metric="watermark_lag_seconds"}` | Wazuh ingest watermark lag in seconds |
| `ai_soc_worker_heartbeat_age_seconds` | Age of the latest AI SOC worker heartbeat |
| `up{job="ai-soc-api"}` | Prometheus scrape health for AI SOC API metrics |
| `ai_soc_component_status_code{component="wazuh_ingest",status="error"}` | AI SOC platform health error state for Wazuh ingest |

---

## Alert Thresholds

| Alert | Expression | Duration | Severity |
|---|---:|---:|---|
| `WazuhIngestBacklogWarning` | `ai_soc_event_queue_pending_events > 250` | `15m` | warning |
| `WazuhIngestBacklogCritical` | `ai_soc_event_queue_pending_events > 1000` | `10m` | critical |
| `WazuhIngestWatermarkLagHigh` | `watermark_lag_seconds > 1800` | `15m` | warning |
| `AiSocWorkerHeartbeatStale` | `ai_soc_worker_heartbeat_age_seconds > 300` | `10m` | critical |
| `AiSocApiMetricsDown` | `up{job="ai-soc-api"} == 0` | `5m` | critical |
| `WazuhIngestComponentError` | `wazuh_ingest error status == 1` | `5m` | critical |

The thresholds are intentionally conservative to avoid noisy notifications during short ingestion bursts.

## Notification Receivers

The initial implementation uses null receivers.

Next receiver options:

| Receiver | Notes |
|---|---|
| SMTP email | Good first enterprise-grade option |
| Telegram | Practical for lab / personal operations |
| Slack / Teams | Better for team collaboration |
| Internal AI SOC webhook | Best product-oriented future option |
| PagerDuty / Opsgenie | Useful later, but too heavy for the current local-first stage |

Recommended next step:

1. validate Alertmanager with null receivers;
2. add one real receiver, preferably SMTP or Telegram;
3. later add an internal AI SOC webhook to surface active alerts inside the platform.

## Operational Notes

Short Wazuh ingestion bursts are expected and should not trigger immediate notifications.

For this reason, all queue-related alerts use a `for:` duration.

The goal is to alert only when backlog, lag, or component failure persists long enough to suggest a real operational issue.
