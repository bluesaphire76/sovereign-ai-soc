# v0.3 Secrets and Configuration Hardening

## Scope

This step validates and hardens local runtime secrets, environment files and security-relevant configuration for the Sovereign AI SOC console.

## Implemented controls

- Runtime `.env` files are excluded from Git.
- Example environment files are provided without real secrets:
  - `.env.example`
  - `frontend/.env.local.example`
- Local runtime environment files should use restrictive permissions:
  - `.env`: `0600`
  - `frontend/.env.local`: `0600`
- Legacy frontend local-auth login route was removed because authentication is handled by the FastAPI backend.
- Weak fallback credentials such as `admin` / `local-dev-secret` must not exist in active authentication paths.
- Next.js `X-Powered-By` header is disabled at application level.
- Nginx also hides proxied `X-Powered-By` headers.
- Systemd overrides should not contain secrets directly.
- Runtime services are bound to `127.0.0.1` and exposed through Nginx only.

## Operational guidance

Before any non-local or demo exposure:

1. Generate a strong `AI_SOC_AUTH_SECRET`.
2. Rotate Wazuh and PostgreSQL passwords.
3. Ensure `.env` and `frontend/.env.local` are not tracked by Git.
4. Keep local runtime files at `0600`.
5. Avoid storing secrets in systemd unit files where possible.
6. Prefer environment files with restricted permissions or a dedicated secret manager for stronger production deployments.

## Verification commands

```bash

git ls-files .env frontend/.env.local
ls -la .env frontend/.env.local
grep -Rni --exclude-dir=node_modules --exclude-dir=.next "local-dev-secret\\|password123\\|admin:admin" .
