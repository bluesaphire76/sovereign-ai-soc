# v0.6 — AI Governance Hardening

Step 13 introduces a shared AI governance foundation for Sovereign AI SOC.

The goal is to standardize how AI-assisted outputs are classified, labeled and reviewed across investigation, detection engineering, remediation planning and executor governance.

## Principles

- Facts require evidence.
- Inferences require rationale.
- Speculation must be labeled.
- Unsupported claims must not be presented as facts.
- Low-confidence output requires human review.
- Fallback-generated output must be visible.
- Simulation and mock/no-op dispatch must not be presented as production execution.
- AI must not bypass human governance.

## Shared Classifications

The governance layer defines four claim classifications:

- `EVIDENCE_BACKED`
- `INFERRED`
- `SPECULATIVE`
- `UNSUPPORTED`

## Safety Labels

The governance layer also defines reusable presentation labels such as:

- `LOW_CONFIDENCE`
- `REQUIRES_HUMAN_REVIEW`
- `SIMULATION_ONLY`
- `EXECUTION_DISABLED`
- `MOCK_ONLY`
- `FALLBACK_GENERATED`

## Step 18 Runtime Safeguards

Step 18 extends the shared governance layer into the remediation intelligence workflow.
AI-generated remediation output now receives a structured `governance` block that is computed deterministically from the normalized plan.

The runtime governance payload includes:

- `status`: `PASSED`, `PASSED_WITH_WARNINGS`, `REQUIRES_REVIEW`, or `BLOCKED`
- `confidence_score`: normalized 0-100 score
- `evidence_coverage`: `HIGH`, `MEDIUM`, `LOW`, or `NONE`
- `human_review_required`: always true for remediation guidance
- `unsupported_claims`
- `assumptions`
- `limitations`
- `policy_warnings`
- `safety_labels`

The assessment is intentionally conservative:

- every remediation recommendation remains advisory
- every remediation recommendation requires human validation
- missing `evidence_basis` creates a visible warning
- fallback output is low-confidence and visibly labeled
- unsupported claims are surfaced rather than hidden
- execution support is blocked
- no remediation or rollback execution is introduced

The Incident Command Room renders this governance assessment in a compact `AI Governance` section so analysts can see confidence, evidence coverage, assumptions, unsupported claims, limitations and policy warnings before using the guidance.

## Scope

The governance layer classifies and presents AI output. It does not execute actions, approve changes, mutate target systems, or bypass remediation approval workflows.

## Non-Goals

This step does not introduce:

- autonomous AI behavior
- remediation execution
- arbitrary command execution
- frontend redesign
- new router logic
- changes to `api.py`
