# v0.7.0 Minimal Loki Observability

## Purpose

This step adds a conservative Loki + Grafana Alloy logging foundation for AI SOC platform troubleshooting.

The purpose is **observability**, not security detection ingestion.

Prometheus tells us what is unhealthy. Loki helps us understand why.

---

## Architecture

```text
Selected platform logs
        ↓
Grafana Alloy
        ↓
Grafana Loki
        ↓
Grafana Explore
```

---

## Design principles

This implementation is intentionally minimal and conservative.

Core principles:

- Platform observability only.
- No duplication of Wazuh or Suricata telemetry.
- No raw security telemetry in Loki.
- No investigation payloads in Loki.
- No AI prompts or AI responses in Loki.
- No HTTP access telemetry in Loki.
- No high-cardinality labels.
- Local-only exposure.
- Short retention.

---

## Collected sources

The initial collection scope is intentionally minimal.

Allowed sources:

- `ai-soc-api` systemd journal logs, filtered to operational lines only.
- `ai-soc-worker` systemd journal logs, filtered to operational lines only.
- `ai-soc-frontend` systemd journal logs, filtered to operational lines only.
- `nginx` and `cloudflared` operational errors/status only.
- Selected observability/support Docker containers:
  - `ai-soc-prometheus`
  - `ai-soc-grafana`
  - `ai-soc-alertmanager`
  - `ai-soc-ntfy-bridge`
  - `ai-soc-loki`
- `ollama` timeout/deadline/ReadTimeout lines only.

---

## Explicit exclusions

The following are intentionally not collected:

- Wazuh alerts.
- Wazuh raw logs.
- Wazuh `/var/ossec` logs.
- Suricata `eve.json`.
- Suricata raw events.
- Security alert payloads.
- Raw incident data.
- Case notes.
- AI prompts.
- AI responses.
- JWT tokens.
- Cookies.
- Authorization headers.
- API request bodies.
- Passwords.
- Secrets.
- Cloudflare tokens.
- ntfy topics.
- HTTP access logs containing IPs, ports, paths or user activity metadata.
- High-cardinality security telemetry.

---

## Privacy and minimization

The Alloy pipeline is intentionally conservative.

It drops:

- sensitive-looking tokens and credentials;
- investigation payload markers;
- raw event/security alert markers;
- HTTP access logs such as `/health`, `/metrics`, `/platform/health`;
- non-timeout Ollama logs;
- very long log lines.

Ollama logs are kept only for timeout/deadline diagnostics.

This keeps Loki focused on runtime troubleshooting rather than user activity, investigation content or security telemetry.

---

## Retention

Loki retention is configured to 7 days.

```yaml
limits_config:
  retention_period: 168h

compactor:
  retention_enabled: true
```

The Loki schema is configured from the v0.7.0 introduction window:

```yaml
schema_config:
  configs:
    - from: 2026-06-01
      store: tsdb
      object_store: filesystem
      schema: v13
      index:
        prefix: index_
        period: 24h
```

The `from` value does not define retention. It defines from which date Loki applies this storage/index schema.

---

## Docker networking

Loki and Alloy run on the same external Docker network used by Grafana:

```text
ai_soc_observability
```

This allows Grafana to reach Loki through Docker DNS:

```text
http://loki:3100
```

Loki and Alloy ports are bound to localhost only:

```text
127.0.0.1:3100
127.0.0.1:12345
```

## Grafana usage

Use Grafana Explore with datasource `Loki`.

Recommended initial queries:

```logql
{job="ai-soc-platform"}
```

```logql
{service="ai-soc-worker"}
```

```logql
{service="ai-soc-api"}
```

```logql
{service="ollama"}
```

```logql
{source="docker"}
```

For this first implementation, use **Grafana Explore**, not Logs Drilldown. Logs Drilldown depends on additional Grafana app/plugin behavior and is outside the scope of this minimal foundation.

## Validation

### Check Loki readiness

```bash
curl -sS http://127.0.0.1:3100/ready
```

Expected:

```text
ready
```

### Check Alloy health

```bash
curl -sS http://127.0.0.1:12345/-/healthy
curl -sS http://127.0.0.1:12345/-/ready
```

Expected:

```text
All Alloy components are healthy.
```

### Check Grafana can reach Loki

```bash
docker exec ai-soc-grafana sh -lc '
wget -qO- http://loki:3100/ready 2>/dev/null || curl -sS http://loki:3100/ready || true
'
```

Expected:

```text
ready
```

### Check Loki sources

```bash
curl -sG http://127.0.0.1:3100/loki/api/v1/label/source/values | jq .
```

Expected sources:

```json
[
  "docker",
  "journal"
]
```

### Check service labels

```bash
curl -sG http://127.0.0.1:3100/loki/api/v1/label/service/values | jq .
```

Expected allowed services may include:

```text
ai-soc-alertmanager
ai-soc-api
ai-soc-grafana
ai-soc-loki
ai-soc-ntfy-bridge
ai-soc-prometheus
ai-soc-worker
ollama
```

### Forbidden source check

```bash
curl -sG http://127.0.0.1:3100/loki/api/v1/label/service/values \
  | jq -r '.data[]?' \
  | grep -Ei 'wazuh|suricata|ossec|eve|postgres|qdrant' \
  && echo "FORBIDDEN SOURCE FOUND" \
  || echo "OK"
```

Expected:

```text
OK
```

### Ollama timeout-only check

```bash
curl -sG http://127.0.0.1:3100/loki/api/v1/query_range \
  --data-urlencode 'query={service="ollama"}' \
  --data-urlencode 'limit=50' \
  | jq -r '.data.result[].values[][1]?' \
  | grep -Eiv 'timeout|ReadTimeout|deadline exceeded|context deadline|timed out' \
  && echo "WARNING: non-timeout Ollama line found" \
  || echo "OK: Ollama timeout-only or empty"
```

Expected:

```text
OK: Ollama timeout-only or empty
```

### HTTP access log drop check

Use a recent time window to avoid matching old logs collected before the final hardening.

```bash
END_NS=$(date +%s%N)
START_NS=$(date -d '90 seconds ago' +%s%N)

curl -sG http://127.0.0.1:3100/loki/api/v1/query_range \
  --data-urlencode 'query={service="ai-soc-api"} |~ "(GET|POST|/health|/metrics|/platform/health)"' \
  --data-urlencode "start=$START_NS" \
  --data-urlencode "end=$END_NS" \
  --data-urlencode 'limit=20' \
  | jq '.data.result | length'
```

Expected:

```text
0
```

### Sensitive pattern check

```bash
for pattern in 'authorization' 'bearer ' 'cookie' 'set-cookie' 'jwt' 'password' 'secret' 'api_key' 'apikey' 'access_token' 'refresh_token' 'prompt' 'response' 'raw_alert' 'raw_event' 'incident_id' 'case_id'; do
  echo "Checking pattern: $pattern"
  curl -sG http://127.0.0.1:3100/loki/api/v1/query_range \
    --data-urlencode "query={job=\"ai-soc-platform\"} |~ \"(?i).*${pattern}.*\"" \
    --data-urlencode 'limit=1' \
    | jq '.data.result | length'
done
```

Expected: all checks should return `0`.

## Start

Start Loki and Alloy with the observability compose override:

```bash
docker compose \
  -f deploy/observability/docker-compose.yml \
  -f deploy/observability/docker-compose.loki.yml \
  up -d loki alloy
```

Restart Grafana to load the Loki datasource:

```bash
docker compose \
  -f deploy/observability/docker-compose.yml \
  -f deploy/observability/docker-compose.loki.yml \
  restart grafana
```

---

## Rollback

Stop Loki and Alloy:

```bash
docker compose \
  -f deploy/observability/docker-compose.yml \
  -f deploy/observability/docker-compose.loki.yml \
  stop alloy loki
```

Remove containers:

```bash
docker compose \
  -f deploy/observability/docker-compose.yml \
  -f deploy/observability/docker-compose.loki.yml \
  rm -f alloy loki
```

Optional volume cleanup:

```bash
docker volume ls | grep loki
docker volume rm <loki_volume_name>
```

Do not delete volumes automatically.

## Notes

This implementation is intentionally conservative.

Loki is used to observe the AI SOC platform runtime only. Wazuh and Suricata remain the security detection sources and are intentionally excluded from Loki collection.

Future extensions may add log-based alerts, dashboards or richer platform diagnostics, but only after validating signal quality and privacy impact.
