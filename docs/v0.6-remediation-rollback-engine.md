# v0.6 Step 15 - Rollback Engine

Step 15 adds a structured rollback readiness engine for the remediation workflow.

The rollback engine helps analysts review whether proposed remediation actions have a viable rollback path before any future operational approval. It remains planning-only and does not execute remediation or rollback actions.

## Boundary

Step 15 does not:

- execute rollback
- execute remediation
- run shell commands
- contact target systems
- mutate users, files, services, firewall rules, hosts or configurations
- add a run or execute control to the UI

The backend response always includes:

- `execution_supported: false`
- `rollback_execution_supported: false`
- `human_approval_required: true`

## Endpoint

```text
GET /incidents/{incident_id}/remediation-rollback-readiness
```

The router is read-only and delegates to `remediation.rollback_engine`. `api.py` only contains the centralized RBAC allowlist entry.

## Engine Flow

```text
remediation intelligence plan
 -> structured remediation plan adapter
 -> per-action rollback readiness
 -> plan-level blockers and warnings
 -> normalized rollback engine response
```

The engine reuses the existing remediation models:

- `RollbackPlan`
- `RollbackAvailability`
- `RollbackStep`
- `RemediationRollbackReadiness`
- `RemediationRollbackReadinessStatus`

## Statuses

The normalized output exposes:

- `READY`
- `CONDITIONAL`
- `NOT_READY`
- `UNKNOWN`

`NOT_READY` is used when rollback is missing, blocked or unavailable. `CONDITIONAL` is used when rollback is partial or requires owner or analyst validation.

## Frontend

Incident Detail includes a compact **Rollback Readiness** section in the AI investigation console. It shows:

- overall status
- execution and rollback execution flags
- per-action rollback status
- rollback risk
- blockers and warnings
- validation notes

The panel is additive and does not block the main incident page load.

## Governance

Rollback readiness is a review artifact. Human approval remains mandatory before any future operational change. This step prepares governance for future safe executor work without adding execution capability.
