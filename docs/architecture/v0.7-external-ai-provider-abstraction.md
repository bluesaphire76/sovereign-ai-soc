# AI SOC v0.7 External AI Provider Abstraction

## Purpose

v0.7 introduces an internal AI provider abstraction for AI SOC while preserving the
platform default:

- local-first
- sovereign by default
- external AI disabled by default
- explicit provider selection
- strict baseline data controls
- auditability for non-local provider calls
- deterministic fallback behavior in existing AI flows

External AI is not the production default. The abstraction is connected to AI
Data Control, provider settings and Health provider visibility.

## Provider Types

The registry supports these provider types:

- `LOCAL_OLLAMA`
- `OPENAI_COMPATIBLE`
- `AZURE_OPENAI_COMPATIBLE`
- `ANTHROPIC_COMPATIBLE`
- `CUSTOM_HTTP_COMPATIBLE`
- `DISABLED`

`LOCAL_OLLAMA` remains the default. `OPENAI_COMPATIBLE` has a concrete adapter.
The other external provider types are configuration-visible and
health/status-visible, but their production adapters are deferred.

## Configuration

Default environment values:

```bash
AI_PROVIDER_DEFAULT=local_ollama
AI_EXTERNAL_PROVIDERS_ENABLED=false
AI_PROVIDER_CONFIG_PATH=storage/config/ai_providers.json
```

Runtime config can be supplied with environment variables or a local JSON file.
Use `storage/config/ai_providers.example.json` as the template. The real
`storage/config/ai_providers.json` file is ignored by git.

External providers require all of the following before any call can occur:

- `AI_EXTERNAL_PROVIDERS_ENABLED=true`
- provider `enabled=true`
- provider fully configured
- selected feature included in the provider feature allowlist
- redaction mode is not `BLOCK_EXTERNAL` or `LOCAL_ONLY`

## Feature Allowlist

External provider allowlists are empty by default. Common feature keys include:

- `incident_triage`
- `incident_ai_analysis`
- `incident_command_brief`
- `case_ai_analysis`
- `detection_quality_how_to_execute`
- `executive_insights`
- `report_support`
- `remediation_explanation`

Provider selection is enforced server-side.

## Redaction Modes

Provider redaction modes:

- `LOCAL_ONLY`: local provider only
- `METADATA_ONLY`: sends only high-level metadata
- `REDACTED_CONTEXT`: sends redacted text/context
- `BLOCK_EXTERNAL`: blocks external calls

The deterministic redaction helper masks tokens, password-like values, private
keys, emails, IP addresses, hostnames, usernames, credentialed URLs and file
paths. It does not use an LLM for redaction.

## Audit Behavior

Every external call or blocked external call writes a security audit event:

- `AI_PROVIDER_EXTERNAL_CALL`
- `AI_PROVIDER_EXTERNAL_CALL_BLOCKED`
- `AI_PROVIDER_TEST`
- `AI_PROVIDER_TEST_BLOCKED`

Audit details include provider metadata, feature, model, redaction mode,
redaction status, character counts, latency, fallback status, outcome and safe
error. Raw prompts, raw responses and secrets are not stored.

## API Endpoints

- `GET /ai-providers`
- `GET /ai-providers/capabilities`
- `GET /ai-providers/health`
- `GET /ai-providers/effective-policy`
- `GET /ai-providers/local-profiles`
- `PATCH /ai-providers/settings`
- `PATCH /ai-providers/{provider_key}/config`
- `POST /ai-providers/{provider_key}/test`

Provider test is admin-only. External provider test sends only a harmless
connectivity prompt when the provider is explicitly enabled and configured.
Provider configuration writes are also ADMIN-only.

## UI Disclosure

The Settings navigation includes `AI Providers`, showing:

- default provider
- external provider global state
- provider status, model and configuration presence
- feature allowlist
- redaction mode
- health and safe test result

Incident AI brief and Detection Quality guidance expose provider metadata such as
provider key, model, external AI usage and redaction status.

## Health Behavior

AI runtime health continues to focus on local Ollama. Provider registry health is
also attached to AI runtime details. External health checks are light and do not
send SOC data.

## Current Limitations

- Only `OPENAI_COMPATIBLE` has a concrete external generation adapter.
- Azure, Anthropic and custom HTTP providers are configuration-visible only.
- Per-feature routing is server-side; workflows do not expose an unrestricted
  end-user provider/model picker.
- Streaming responses are not implemented.
- Cost/token tracking is limited to provider usage metadata when returned.
- API keys remain environment-only and are never written by the GUI.

## Relationship To AI Data Control

The [AI Data Control policy](v0.7-ai-data-control-policy.md) is authoritative
for feature, role, provider and data exposure. Provider enablement alone is
never sufficient to send a request externally.

The provider layer does not expose MCP tools or external remediation
capabilities.
