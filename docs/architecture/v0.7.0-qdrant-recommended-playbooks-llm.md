# v0.7.0 Recommended Playbooks LLM Synthesis

## Purpose

Recommended Playbooks now uses the configured AI provider to synthesize the
incident-specific Qdrant retrieval results into structured analyst guidance.

The responsibilities remain separated:

- Qdrant retrieves playbook sections and optional historical context.
- The LLM explains why the retrieved playbooks apply and proposes analyst checks.
- Deterministic code validates titles, output structure and fallback behavior.
- The human analyst decides severity, containment, remediation and closure.

## Runtime Path

The existing endpoints are unchanged:

- `GET /incidents/{incident_id}/recommended-playbooks`
- `GET /cases/{case_id}/recommended-playbooks`

The endpoints still return the existing `recommendations` card list. Additional
backward-compatible fields include:

- `selection_summary`
- `generated_markdown`
- `generation`
- `generation_limitations`

Each recommendation may also include:

- `evidence_to_collect`
- `false_positive_checks`
- `escalation_criteria`
- `containment_remediation_guidance`
- `closure_considerations`
- `generation_source`

## Prompt Boundaries

The LLM prompt separates:

1. current incident or case facts;
2. retrieved playbook context;
3. similar historical incidents, when available;
4. governance and human approval constraints;
5. output requirements.

Only exact retrieved playbook titles are accepted. Unknown LLM-generated titles
are discarded.

For incident recommendations, the existing Similar Incidents service supplies up
to three supporting historical observations. Historical outcomes are labeled as
validation patterns and cannot classify the current incident.

Case recommendations do not add historical incident context because no equivalent
case-similarity service is currently exposed.

## Fallback

If the AI provider times out, fails, returns an empty response or emits invalid
JSON, the endpoint returns deterministic structured guidance derived from the
retrieved playbook metadata and category.

The fallback includes:

- why the playbook was selected;
- immediate analyst checks;
- evidence to collect;
- false-positive checks;
- escalation criteria;
- approval-required containment or remediation options;
- closure considerations.

The endpoint does not return an empty panel when Qdrant supplied usable playbook
context.

## Configuration

The optional timeout is:

```env
RECOMMENDED_PLAYBOOKS_LLM_TIMEOUT_SECONDS=45
```

## Expanded retrieval domains

Recommended Playbooks metadata inference and deterministic ranking support:

- Linux host activity;
- Windows Event Log and optional Sysmon enrichment;
- Suricata network detections;
- DNS anomalies;
- malware and suspicious execution;
- data exfiltration and unusual outbound activity;
- governance decisions requiring human review.

The expanded signal catalog is maintained in
`playbook_retrieval_catalog.py`. It provides metadata candidates only; it does
not classify incidents or authorize actions.

After changing playbook content, run:

```bash
PYTHONPATH=. .venv/bin/python scripts/reindex_qdrant_playbooks.py --dry-run
PYTHONPATH=. .venv/bin/python scripts/reindex_qdrant_playbooks.py --apply
PYTHONPATH=. .venv/bin/python scripts/validate_qdrant_playbook_expansion.py
```

The validation checks all expanded playbook payload metadata and representative
Linux, Windows, Suricata, DNS, Malware, Exfiltration and Governance retrieval
scenarios.

If omitted, the default is 45 seconds. The existing provider routing and local
Ollama fast-profile fallback remain active.

## Offline Validation

The validation script does not call a live LLM:

```bash
PYTHONPATH=. .venv/bin/python scripts/validate_recommended_playbooks_llm_prompt.py
```

It covers:

- SSH brute force;
- SSH success after failures;
- sudo privilege escalation;
- Suricata port scan;
- DNS tunneling;
- weak or broad retrieval context.
