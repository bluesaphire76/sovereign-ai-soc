# AI SOC v0.7 AI Data Control Policy

## Purpose

v0.7 adds a server-side policy engine in front of AI provider calls. The
default remains local-first and sovereign:

- local Ollama is allowed by default
- local llama.cpp is supported as `LOCAL_LLAMA_CPP` when explicitly enabled
- external AI is disabled by default
- every AI request receives a policy decision before execution
- secrets and credentials are deterministically redacted for local and external
  providers
- external providers receive only metadata, redacted context or admin-only full
  context when explicitly allowed
- decisions are recorded through security audit events without storing raw
  prompts or raw SOC telemetry

## Runtime Configuration

```bash
AI_DATA_POLICY_CONFIG_PATH=storage/config/ai_data_control_policy.json
AI_DATA_POLICY_DEFAULT_MODE=LOCAL_ONLY
AI_EXTERNAL_DEFAULT_POLICY=EXTERNAL_AI_DISABLED
AI_POLICY_AUDIT_ENABLED=true
AI_POLICY_PAYLOAD_PREVIEW_ENABLED=true
AI_POLICY_STORE_PAYLOAD_HASH=true
AI_POLICY_STORE_REDACTED_PREVIEW=false
```

Use `storage/config/ai_data_control_policy.example.json` as a starting point.
The runtime file `storage/config/ai_data_control_policy.json` is ignored by git.

## OpenRouter

OpenRouter is available as an OpenAI-compatible provider:

```bash
AI_OPENROUTER_ENABLED=false
AI_OPENROUTER_BASE_URL=https://openrouter.ai/api/v1
AI_OPENROUTER_MODEL=
AI_OPENROUTER_API_KEY=
AI_OPENROUTER_FEATURE_ALLOWLIST=
AI_OPENROUTER_REDACTION_MODE=BLOCK_EXTERNAL
```

The API key stays in `.env`. The GUI can change provider state, model, timeout,
max tokens, allowlist and redaction mode at runtime, but it does not store API
keys in JSON config.

## Policy Modes

- `EXTERNAL_AI_DISABLED`: deny external calls for the feature.
- `LOCAL_ONLY`: allow local execution only.
- `METADATA_ONLY`: external providers receive only safe request metadata.
- `REDACTED_CONTEXT`: external providers receive deterministic redacted context.
- `FULL_CONTEXT_ADMIN_ONLY`: admin-only external full context with secrets and
  credentials still redacted.
- `CUSTOM_ALLOWLIST`: explicit role/provider/data-class allowlist mode.
- `FEATURE_DISABLED`: deny the feature.

## Redaction

The engine emits deterministic markers:

- `[REDACTED_SECRET]`
- `[REDACTED_CREDENTIAL]`
- `[REDACTED_PERSONAL_DATA]`
- `[REDACTED_IP]`
- `[REDACTED_HOST]`
- `[REDACTED_RAW_TELEMETRY]`
- `[REDACTED_FIELD]`

Raw prompts are not stored in audit events. By default, audit details include a
payload hash, mode, provider, feature, action, decision reason and redaction
counts.

## GUI

Settings includes:

- `AI Providers`: runtime provider controls, OpenRouter model selection and
  local LLM profile visibility.
- `AI Data Control`: policy matrix, admin editor, redaction/evaluation preview
  and recent policy decisions.

Admin users can edit policies and provider settings with a required reason.
Analysts can view policy state and run previews. Viewers receive read-only
policy and decision visibility.

## Local LLM Profiles

The provider page shows the three configured local profiles:

- `fast`
- `standard`
- `quality`

The page displays the model, routing, timeout, keep-alive, last-used profile and
Ollama loaded-model status. This is intended to make VRAM constraints visible
when only one model profile should be resident at a time.

## Decision Boundary

AI Data Control decides whether and how a provider request may be prepared. It
does not authorize incident status, severity, Detection Control, remediation or
case-closure changes. Those workflows retain separate RBAC, lifecycle and
approval checks.
