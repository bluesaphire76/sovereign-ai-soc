# v0.7.0 Qdrant Playbook Metadata Indexing

## Purpose

This step makes the Qdrant knowledge-base playbook path metadata-aware for the Recommended Playbooks workflow.

Qdrant remains a semantic retrieval layer. It does not generate analyst-facing conclusions, decide severity, approve remediation, close cases, suppress alerts or replace RBAC, audit, deterministic lifecycle rules or human review.

## What Changed

Playbook Markdown files under `knowledge_base/playbooks/` are parsed recursively and indexed as section-level chunks.

Each playbook chunk preserves the existing knowledge-base payload fields:

- `source_type`
- `source`
- `text`
- `chunk_index`
- `content_hash`

It also adds playbook metadata payload fields:

- `doc_type`
- `kb_type`
- `title`
- `domain`
- `playbook_source`
- `incident_types`
- `severity_hint`
- `mitre_tactics`
- `mitre_techniques`
- `applicability`
- `not_applicable_when`
- `recommended_for_pages`
- `tags`
- `section`
- `section_order`
- `file_path`
- `content_kind`
- `content_preview`

The existing `source` field remains the file path for compatibility with current retrieval and index-status behavior. The `source` value from YAML front matter is stored as `playbook_source`.

## Front Matter

Each playbook can include YAML front matter:

```yaml
---
title: SSH Brute Force Investigation Playbook
type: playbook
domain: authentication
source: wazuh
incident_types:
  - ssh_bruteforce
severity_hint:
  - medium
  - high
mitre_tactics:
  - Credential Access
mitre_techniques:
  - T1110
applicability:
  - Multiple failed SSH login attempts
not_applicable_when:
  - Known vulnerability scanner
recommended_for_pages:
  - recommended_playbooks
  - ai_analysis
  - incident_detail
tags:
  - ssh
  - brute-force
---
```

Missing fields are reported by dry-run but do not crash indexing. Safe defaults are used for title, type, domain, source, recommended pages and tags where possible.

The raw YAML front matter is not embedded into the Qdrant text body.

## Section Chunking

Playbooks are chunked by Markdown H2 sections such as:

- `Purpose`
- `When to Use`
- `Detection Signals`
- `Initial Triage`
- `Evidence to Collect`
- `Investigation Steps`
- `Correlation Checks`
- `False Positive Conditions`
- `Escalation Criteria`
- `Containment Actions`
- `Remediation Actions`
- `Closure Criteria`
- `Analyst Notes`

If a section is too long, it is split while preserving `section`, `section_order` and `section_chunk_index`.

Files without H2 sections are still safely chunked and reported during dry-run.

## Safe Dry-Run and Apply

Use dry-run to preview what would be indexed:

```bash
PYTHONPATH=. .venv/bin/python scripts/reindex_qdrant_playbooks.py --dry-run
```

Dry-run reports:

- target collection;
- collection availability;
- playbook files discovered;
- files with valid front matter;
- files missing required metadata;
- section/chunk count;
- skipped and excluded files;
- sample payloads.

Use apply to refresh only playbook points:

```bash
PYTHONPATH=. .venv/bin/python scripts/reindex_qdrant_playbooks.py --apply
```

Apply deletes only existing playbook knowledge-base points identified by playbook payload metadata or `knowledge_base/playbooks/` source paths, then upserts the current playbook section chunks.

It does not recreate the Qdrant collection and does not delete:

- `historical_incident` memory;
- `detection_control` memory;
- `case_closure` memory;
- similar incident memory derived from incident records.

## Recommended Playbooks Retrieval

Recommended Playbooks retrieval now first requests Qdrant chunks where:

- `source_type = knowledge_base`
- `doc_type = playbook`
- `content_kind = playbook_section`

It then prefers chunks whose `recommended_for_pages` includes `recommended_playbooks`.

If too few results are returned, retrieval falls back to general knowledge-base semantic search. The existing response shape remains compatible, and the LLM remains responsible for incident/case analysis and analyst-facing narrative generation.

## Validation

After apply, inspect Qdrant payloads:

```bash
curl -s -X POST http://127.0.0.1:6333/collections/security_kb/points/scroll \
  -H 'Content-Type: application/json' \
  -d '{
    "limit": 5,
    "with_payload": true,
    "with_vector": false,
    "filter": {
      "must": [
        {"key": "doc_type", "match": {"value": "playbook"}}
      ]
    }
  }' | jq
```

Expected payloads should include fields such as `title`, `domain`, `playbook_source`, `incident_types`, `recommended_for_pages`, `section`, `file_path`, `tags` and `content_kind`.
