# v0.6 Step 1 - Structured Investigation Domain Model

This step introduces the backend/domain-model foundation for the v0.6 roadmap:

**Iterative AI Investigation and Human-Approved Remediation**

The implementation is intentionally limited to typed investigation data structures, deterministic validation helpers and fallback factories. It does not implement the investigation engine, remediation engine, execution workflow, persistence migration or frontend redesign.

## Purpose

The goal is to move future AI investigation output away from unstructured text and toward a reusable, evidence-backed, auditable model that can later support:

- investigation sessions
- evidence-backed hypotheses
- confidence evolution
- iterative evidence expansion
- analyst feedback
- investigation memory
- human-approved remediation planning

## Module Layout

The new package is:

```text
investigation_ai/
  __init__.py
  models.py
  validators.py
  factory.py
```

No business logic was added to `api.py`, and no new API endpoint was introduced in this step.

## Model Overview

The domain model includes:

- `InvestigationSession`
- `InvestigationBrief`
- `EvidenceReference`
- `ConfidenceAssessment`
- `InvestigationHypothesis`
- `RecommendedCheck`
- `RecommendedAction`
- `InvestigationFinding`
- `InvestigationLimitation`

Stable enums define session status, evidence type, evidence strength, hypothesis status, finding type, confidence level, claim classification, check priority, action category and approval requirement.

## Relationship with Existing AI Analysis

Existing Incident AI Analysis, AI Command Brief, case analysis and report enrichment remain unchanged.

The new model is a foundation for future steps to map AI-generated or deterministic investigation output into structured records that are easier to validate, persist, render and audit.

## Validation Rules

`investigation_ai/validators.py` provides deterministic validation for:

- hypotheses requiring supporting evidence or explicit missing evidence
- unsupported claims being flagged before they become findings
- operational recommended actions requiring human approval
- recommended actions remaining non-executable in Step 1
- confidence score normalization between 0 and 100

No LLM call is made by the validators.

## Fallback Factories

`investigation_ai/factory.py` provides:

- `create_empty_investigation_brief`
- `create_fallback_investigation_brief`

These helpers keep future workflows able to return valid structured output even when a full investigation engine is unavailable.

## Remediation Boundary

No remediation execution is implemented in this step.

The model includes `RecommendedAction.execution_supported`, but it defaults to `False`, and validators flag executable actions as invalid for Step 1. Future remediation steps must add explicit governance, approval, audit and execution boundaries before any operational action is supported.

## Frontend Scope

No frontend changes were made. A TypeScript mirror of the investigation schema can be added in a later step when UI rendering or API payloads are introduced.
