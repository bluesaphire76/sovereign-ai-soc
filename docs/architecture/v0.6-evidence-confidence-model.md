# v0.6 Evidence-Backed Reasoning and Confidence Model

## Purpose

Step 3 moves the investigation layer from AI-generated summaries toward evidence-backed investigative reasoning.

The platform now treats investigation output as structured decision support. Important claims must be linked to evidence, missing evidence must be explicit, and confidence is calculated deterministically rather than accepted from LLM wording.

Confidence is not purely LLM-generated.

Unsupported AI claims must not be represented as facts.

## Modules

- `investigation_ai/evidence.py` normalizes incident, raw event, security alert, correlation, MITRE and existing AI analysis references into stable evidence objects.
- `investigation_ai/reasoning.py` performs deterministic evidence aggregation, missing-evidence identification, conservative contradiction detection and claim classification.
- `investigation_ai/confidence.py` calculates deterministic `ConfidenceAssessment` objects using scoring factors.
- `investigation_ai/engine.py` consumes those modules when producing `InvestigationBrief` output.

No business logic is added to `api.py`.

## Deterministic Confidence

Confidence is calculated from observable signals:

- supporting evidence count and strength
- multiple correlated events
- repeated or matched patterns
- MITRE mapping
- IOC-like evidence
- timeline availability
- missing evidence
- contradictory evidence
- noisy operational patterns
- unsupported claim penalties

Each confidence assessment includes:

- normalized score
- confidence level
- rationale
- positive signals
- negative signals
- missing evidence
- contradictory evidence
- scoring factors used

## Evidence-Backed Hypotheses

Hypotheses now carry:

- supporting evidence references
- contradictory evidence references
- missing evidence
- deterministic confidence
- rationale
- claim classification

A hypothesis without supporting evidence cannot become strongly confident unless explicit evidence is later attached. Missing evidence remains visible so the analyst can decide the next investigation step.

## Unsupported Claim Handling

Unsupported content is downgraded through deterministic governance:

- `UNSUPPORTED` hypotheses and findings are reclassified as speculative decision-support content.
- unqualified certainty language is softened.
- unsupported claims apply a confidence penalty.
- limitations are added to the brief for analyst review.

This prevents unsupported AI claims from appearing as hard facts.

## Conservative Contradiction Detection

Step 3 introduces structured skepticism, not perfect detection.

The reasoning layer can identify dampening or contradictory context such as:

- false-positive or suppressed/noisy operational context
- inconsistent timeline ordering
- hypotheses that require missing evidence, such as brute force without successful login verification
- privilege escalation hypotheses without sudo or escalation logs
- malware hypotheses without process tree or file hash evidence

Contradictions reduce confidence and remain visible for analyst review.

## Missing Evidence Suggestions

The engine now suggests missing evidence such as:

- raw event payload
- normalized security alert record
- structured correlation summary
- event timeline
- successful login verification
- sudo or privilege escalation logs
- process tree and file hash evidence
- DNS and outbound network activity

This prepares future iterative investigation loops without implementing autonomous retrieval in this step.

## Relationship With Future v0.6 Work

This step creates the reasoning foundation for later capabilities:

- iterative evidence expansion
- investigation memory
- analyst feedback loops
- confidence evolution
- cross-incident intelligence
- human-approved remediation planning

## Non-Goals

Step 3 does not implement:

- remediation execution
- autonomous investigation loops
- cross-incident campaign analysis
- investigation persistence
- SOAR automation
- frontend redesign
- new API endpoints

The investigation model remains analyst-centric and human-in-the-loop.
