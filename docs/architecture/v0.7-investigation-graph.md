# v0.7 Investigation Graph

## Purpose

The Investigation Graph is a read-only relationship view for incidents and
cases. It connects stored evidence and workflow records so analysts can see
which hosts, users, IP addresses, processes, files, packages, MITRE
techniques, detection controls, alerts and AI hypotheses are related.

The graph is an investigation aid. It does not create incidents, infer a final
root cause, change severity or authorize remediation.

## User Experience

Incident Detail and Case Detail contain a collapsible `Investigation Graph`
section. The UI supports:

- incident-scoped and case-scoped graphs;
- node and edge type filters;
- timeline, AI, detection-rule and suppression context;
- optional raw-event inclusion where the role allows it;
- graph summaries and quality warnings;
- evidence references on nodes and edges;
- bounded depth and result sizes.

## API

```text
GET /investigation-graph/capabilities
GET /investigation-graph/incidents/{incident_id}
GET /investigation-graph/incidents/{incident_id}/summary
GET /investigation-graph/cases/{case_id}
GET /investigation-graph/cases/{case_id}/summary
```

Graph query controls:

| Parameter | Default | Limit |
|---|---:|---:|
| `depth` | `1` | `2` |
| `include_raw_events` | `false` | boolean |
| `include_timeline` | `true` | boolean |
| `include_ai` | `true` | boolean |
| `include_detection_rules` | `true` | boolean |
| `include_suppression` | `true` | boolean |
| `limit_nodes` | `80` | `200` |
| `limit_edges` | `160` | `400` |

## Data Sources

The graph uses existing, linked records including:

- incidents and cases;
- security alerts, raw events and event aggregates;
- incident and case timelines;
- hosts, users, source/destination IP addresses and other extracted entities;
- MITRE techniques;
- Detection Control rules, exceptions and noise suppression;
- investigation sessions, snapshots and hypotheses;
- case AI analysis.

No unbounded global scan is performed. Internal reads and public output are
capped, and truncation is reported through graph warnings.

## Redaction and RBAC

All authenticated roles can read the graph through the central RBAC layer.
Sensitive metadata keys are removed. `VIEWER` receives redacted raw-event
metadata even when raw events are requested; ADMIN and ANALYST retain the
role-appropriate evidence view.

Frontend hiding is not the security boundary. Backend RBAC and server-side
metadata sanitization remain authoritative.

## Interpretation Rules

- An edge means the records share an observed or stored relationship.
- Edge weight and evidence count describe support density, not certainty.
- AI hypothesis nodes are hypotheses, not evidence-backed facts by themselves.
- A sparse graph can be correct when little linked evidence exists.
- A truncated graph must be narrowed with filters before drawing conclusions.
- The graph does not replace the Advanced Incident Timeline, raw evidence or
  analyst validation.
