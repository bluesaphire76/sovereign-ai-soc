# v0.6 Investigation Retrieval Foundation

## Purpose

Step 4 introduces the first controlled retrieval layer for v0.6 iterative investigation.

This is not an autonomous investigation agent.

Retrieval expansion is bounded and controlled.

The goal is to let the investigation engine identify missing evidence, request additional context, merge retrieved evidence, and refine the investigation brief without creating recursive agent behavior or operational execution.

## Architecture

The Step 4 implementation is contained in `investigation_ai/`:

- `retrieval.py` defines retrieval request/result models, retrieval limits, prioritization, bounded execution and retrieval audit data.
- `expansion.py` identifies missing evidence from a brief, runs one enrichment pass, merges retrieved evidence and refines recommended checks.
- `engine.py` can optionally run a single bounded enrichment pass before returning a refined deterministic brief.

No business logic is added to `api.py`.

No frontend changes are introduced in Step 4.

## Retrieval Request Model

The retrieval layer supports structured request types:

- `SAME_HOST_EVENTS`
- `SAME_USER_EVENTS`
- `SAME_IP_EVENTS`
- `SAME_RULE_EVENTS`
- `RELATED_ALERTS`
- `MITRE_RELATED_EVENTS`
- `TIMELINE_EXPANSION`
- `PACKAGE_ACTIVITY`
- `SUDO_ACTIVITY`
- `AUTH_ACTIVITY`
- `DNS_ACTIVITY`
- `NETWORK_ACTIVITY`

Each request captures:

- retrieval type
- reason
- priority
- requested missing evidence
- entity filters
- source systems
- bounded result count
- retrieval depth

## Retrieval Boundaries

Retrieval uses explicit limits:

- max retrieval depth
- max retrieved objects
- max requests
- max timeline expansion
- max related entities
- timeout threshold

The engine only performs one enrichment pass in Step 4. There is no recursive investigation loop.

## Enrichment Flow

The controlled flow is:

1. Generate an initial structured `InvestigationBrief`.
2. Collect missing evidence from confidence, hypotheses and limitations.
3. Build prioritized retrieval requests.
4. Execute bounded retrieval against provided candidate evidence or a future fetcher.
5. Merge retrieved evidence with existing evidence.
6. Regenerate a refined deterministic brief.
7. Recalculate confidence and add retrieval review checks.

## Auditability

Every retrieval result records:

- request id
- retrieval type
- status
- evidence returned
- requested and completed timestamps
- duration
- limits applied
- failures or skipped reason

This prepares future replay, persistence and governance without adding storage in Step 4.

## Failure Handling

Retrieval failures are non-blocking.

The layer handles:

- timeouts
- exceptions from future fetchers
- depth-limit skips
- empty candidates
- object/request limits

If retrieval cannot enrich the investigation, the initial brief remains valid.

## Relationship With Future Iterative Investigation

This foundation enables later steps:

- persisted investigation sessions
- evidence retrieval from database-backed providers
- analyst-approved additional evidence requests
- confidence evolution over multiple passes
- structured investigation replay

Step 4 deliberately does not implement autonomous loops, cross-incident intelligence or remediation execution.

## Limitations

Step 4 does not add:

- API endpoints
- frontend rendering
- database persistence for retrieval sessions
- recursive autonomous investigation
- remediation execution
- SOAR or active response

The retrieval layer is a bounded orchestration foundation for analyst-centric investigation.
