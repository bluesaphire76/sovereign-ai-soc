# v0.6 Remediation Planning Foundation

## Purpose

Step 8 introduces the foundation for human-approved AI remediation planning.

The platform can now represent a structured remediation plan with proposed actions, evidence, impact, risk, rollback considerations and approval requirements.

Step 8 does not implement remediation execution.

AI may propose remediation plans, but all operational actions require human approval before any future execution layer can act.

## Architecture

The implementation lives in `remediation/`:

- `models.py` defines remediation plans, actions, targets, approvals, risk assessments, impact assessments, rollback plans, pre-checks, post-checks and validation results.
- `risk.py` provides deterministic risk and approval classification.
- `rollback.py` creates rollback planning artifacts for future review.
- `planner.py` creates planning-only remediation plans from incident and investigation context.
- `validators.py` enforces non-executable defaults, approval requirements, forbidden action rules and command preview boundaries.

No business logic is added to `api.py`.

No frontend changes are introduced in Step 8.

## Planning vs Execution Boundary

The remediation module is planning-only.

It does not:

- execute shell commands
- connect to hosts
- run SSH
- trigger Wazuh active response
- modify firewall rules
- disable users
- stop or restart services
- quarantine files
- kill processes
- isolate hosts
- modify production rules or configuration

Every model-level execution flag defaults to `false` and is forced back to `false` by validators/model guards.

## Approval Requirement Model

Supported approval levels:

- `NONE`
- `ANALYST_APPROVAL`
- `ADMIN_APPROVAL`
- `SECURITY_LEAD_APPROVAL`
- `FORBIDDEN_BY_DEFAULT`

Low-impact workflow actions such as ticket creation or owner notification may be modeled with no approval requirement.

Operational actions require approval.

High-impact or destructive actions are forbidden by default in Step 8.

Examples of forbidden-by-default model actions:

- disable user
- stop service
- quarantine file
- kill process
- isolate host

These are represented only so future governance workflows can reason about them safely.

## Risk Model

Risk is calculated deterministically from:

- remediation action type
- target criticality
- rollback availability
- investigation confidence
- supporting evidence count

Risk levels:

- `INFORMATIONAL`
- `LOW`
- `MEDIUM`
- `HIGH`
- `CRITICAL`

The risk model is intentionally conservative. Missing rollback, unknown confidence and missing evidence increase risk.

## Rollback Model

Every plan and action must include rollback consideration.

Rollback availability can be:

- `FULL`
- `PARTIAL`
- `UNAVAILABLE`
- `NOT_APPLICABLE`

If rollback is unavailable, the plan records limitations and risk is increased.

Rollback planning is advisory in Step 8. There is no automated rollback engine.

## Command Preview Policy

`command_preview` may contain human-readable proposed command-like text for review.

Command previews are never executable by the system in Step 8.

Validators enforce:

- `execution_supported = false`
- `command_preview_is_executable = false`
- shell-like operators are treated as review warnings, not execution instructions

Command previews must be treated as analyst review content only.

## Relationship With Investigation Engine

The planner can consume:

- incident context
- structured investigation brief
- evidence references
- investigation recommended actions
- detection engineering recommendations where useful

The remediation plan does not overload `InvestigationBrief`. It remains a separate planning artifact that can later be persisted, reviewed or rendered.

## Human-in-the-loop Governance

The intended future workflow is:

```text
AI investigates
 -> AI proposes remediation plan
 -> AI explains evidence, impact, risk and rollback
 -> human reviews
 -> human approves, rejects or modifies
 -> future controlled executor may act only on approved and whitelisted actions
```

Step 8 implements only the planning foundation through the first four stages.

## Limitations

Step 8 does not add:

- remediation approval persistence
- execution audit trail
- safe executor
- dry-run executor
- Wazuh active response
- SOAR workflows
- frontend remediation UI
- autonomous remediation

The foundation is designed so future execution work can be added behind explicit approval, audit and whitelist boundaries without changing the existing investigation model.
