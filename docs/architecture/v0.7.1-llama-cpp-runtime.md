# v0.7.1 llama.cpp Runtime

## Purpose

v0.7.1 adds a governed llama.cpp local runtime path alongside the existing
Ollama default. The goal is to let operators evaluate local GGUF-backed model
profiles without changing the product boundary: AI remains analyst decision
support, not an autonomous response engine.

## Runtime Role

llama.cpp is represented as the `LOCAL_LLAMA_CPP` provider type. It is local,
disabled by default, and routed through the same provider registry, AI Data
Control, Health and AI Providers visibility used by the rest of the AI layer.

Ollama remains the default local provider. llama.cpp is an optional local path
for deployments that want a router-managed GGUF runtime and explicit profile
switching.

## Provider Type and Boundary

`LOCAL_LLAMA_CPP` is a local provider, not an external provider. It does not
require external-provider global enablement and should not be described as an
OpenRouter or SaaS dependency.

The provider still follows the same governance model:

- provider enablement is explicit;
- model/profile metadata is visible in Health and AI Providers;
- fallback behavior is surfaced when generation is unavailable;
- AI Data Control remains authoritative for feature, role, provider and data
  exposure decisions;
- outputs remain advisory and human-reviewed.

## Environment Configuration

The committed `.env.example` exposes the runtime foundation:

```bash
LLAMA_CPP_ENABLED=false
LLAMA_CPP_ROUTER_ENABLED=true
LLAMA_CPP_BASE_URL=http://127.0.0.1:8081
LLAMA_CPP_API_BASE_URL=http://127.0.0.1:8081/v1
LLAMA_CPP_API_KEY=no-key

LLAMA_CPP_FAST_MODEL=ai-soc-fast
LLAMA_CPP_STANDARD_MODEL=ai-soc-standard
LLAMA_CPP_QUALITY_MODEL=ai-soc-quality

LLAMA_CPP_AUTO_PROFILE_SWITCH=true
LLAMA_CPP_EXCLUSIVE_MODEL=true
LLAMA_CPP_PROFILE_SWITCH_LOCK=/tmp/ai-soc-llama-cpp-profile.lock
```

`LLAMA_CPP_API_BASE_URL` is the local OpenAI-compatible generation surface.
`LLAMA_CPP_BASE_URL` is the router/control surface used for health and managed
profile visibility.

## Profiles

The router profile names are stable:

| Profile | Model environment variable | Typical role |
|---|---|---|
| `fast` | `LLAMA_CPP_FAST_MODEL` | Low-latency guidance and short summaries. |
| `standard` | `LLAMA_CPP_STANDARD_MODEL` | Default local analysis profile. |
| `quality` | `LLAMA_CPP_QUALITY_MODEL` | Higher-latency reasoning where configured. |

The platform is designed to load the model profile needed by a task, not to
keep every configured model hot at once.

## Router Health and Model Switching

`LLAMA_CPP_ROUTER_ENABLED` controls router-aware health and profile metadata.
`LLAMA_CPP_AUTO_PROFILE_SWITCH` allows the provider path to request the needed
profile. `LLAMA_CPP_EXCLUSIVE_MODEL` and `LLAMA_CPP_PROFILE_SWITCH_LOCK` protect
one-model-at-a-time switching for constrained local hosts.

Health and metrics expose router reachability, model availability, loaded
state and stable model-status labels where the router reports them.

## Fallback Behavior

If llama.cpp is disabled, unreachable, missing a configured model or unable to
generate safely, the workflow should fall back through the existing governed AI
fallback paths. Fallback text must remain clear to the analyst and should not
pretend a model produced guidance when it did not.

## UI and Health Visibility

The AI Providers and Health views distinguish:

- provider type (`LOCAL_OLLAMA`, `LOCAL_LLAMA_CPP`, external-compatible types);
- provider/model/profile metadata;
- router reachability and loaded model state where available;
- fallback usage and safe error metadata;
- external-provider global state separately from local runtime health.

## Security and Data-Control Boundary

llama.cpp is local, but it is still part of the governed AI surface. It does
not bypass RBAC, Security Audit, AI Data Control, deterministic correlation,
Detection Control lifecycle, remediation approval or analyst validation.

Qdrant context sent to any local model remains advisory semantic memory. It is
not an authority for severity, suppression, closure or remediation approval.

## Validation Commands

Useful checks:

```bash
.venv/bin/python scripts/check_api_composition_root.py
./ai-soc validate
./ai-soc validate-runtime
```

When runtime services are available, also review:

- Health page provider and router metadata;
- AI Providers page local profile and fallback details;
- Prometheus metrics for llama.cpp router/model state.

## Known Limitations

- llama.cpp is disabled by default in `.env.example`.
- Operators must provide and manage the local router/runtime and GGUF models.
- The repository does not download models automatically.
- Profile switching depends on the configured local router behavior.
- AI output remains advisory even when generated fully locally.
