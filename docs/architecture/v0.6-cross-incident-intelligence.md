# v0.6 Cross-Incident Intelligence Foundation

## Purpose

Step 6 introduces explainable cross-incident intelligence for Sovereign AI SOC.

The platform now has a deterministic foundation for identifying similar incidents, recurring entities and repeated investigative patterns. This is contextual analyst support, not autonomous threat hunting.

## Architecture

The implementation is contained in `investigation_ai/`:

- `similarity.py` builds incident similarity profiles, calculates deterministic similarity scores and explains why two incidents are considered related.
- `intelligence.py` orchestrates bounded historical context analysis, recurring entity tracking, recurring pattern detection and brief enrichment.
- `persistence.py` can persist similarity metadata for replay and governance.

No business logic is added to `api.py`.

No frontend changes are introduced in Step 6.

## Deterministic Similarity Scoring

Similarity scoring is deterministic and explainable. It uses conservative signals such as:

- MITRE technique overlap
- same source IP
- same destination IP
- same user
- same host or agent
- same rule or detector
- timeline proximity
- evidence text similarity
- hypothesis similarity

Each signal contributes an explicit weight and is recorded in the similarity output.

## Similarity Explanation

Every match includes:

- score
- strength
- scoring signals
- overlapping values
- matched MITRE techniques
- matched rule identifiers
- matched entity values
- rationale

The platform must always be able to answer:

```text
Why this incident is considered related.
```

## Boundaries and Safeguards

Similarity analysis is bounded by:

- maximum related incidents
- maximum time window
- maximum entity expansion
- maximum similarity depth
- timeout threshold
- minimum similarity score

There is no uncontrolled graph traversal and no recursive historical expansion in Step 6.

## Historical Context Enrichment

The investigation engine can optionally enrich an `InvestigationBrief` with historical similarity context.

When enabled, the brief may receive:

- a contextual finding for similar historical incidents
- a recommended check to review related incidents
- a limitation explaining that similarity is contextual, not proof
- a conservative confidence adjustment
- a next investigation step for historical review

This enrichment is deterministic and remains analyst-controlled.

## Recurring Entity Tracking

The intelligence layer tracks recurring entities such as:

- hosts
- users
- source IPs
- destination IPs
- rule IDs
- MITRE techniques

Entity recurrence is derived from similarity matches and remains explainable.

## Recurring Pattern Tracking

The layer can identify recurring patterns such as:

- MITRE overlap
- rule overlap
- entity overlap
- timeline proximity
- investigation pattern overlap

These patterns support analyst context and future replay, not autonomous conclusions.

## Persistence

Similarity metadata can be persisted in `investigation_similarity_history`.

Stored metadata includes:

- current incident id
- related incident id
- similarity score
- similarity strength
- scoring signals
- recurring entities
- recurring patterns
- rationale

This keeps cross-incident context replay-friendly and auditable.

## Limitations

Step 6 does not add:

- autonomous threat hunting
- black-box clustering
- machine-learning similarity pipelines
- campaign detection
- frontend UI
- new API endpoints
- remediation execution
- SOAR workflows

Cross-incident intelligence is contextual decision support. Similarity does not prove common cause, shared actor or campaign activity without analyst validation and supporting evidence.
