# v0.6 Investigation Engine Foundation

## Purpose

Step 2 introduces the first backend investigation engine layer for the v0.6 roadmap: **Iterative AI Investigation & Human-Approved Remediation**.

The engine converts incident and evidence context into a typed `InvestigationBrief`. It is designed as a safe foundation for future iterative investigation workflows, not as an autonomous SOC agent.

## Architecture

The implementation is contained in `investigation_ai/`:

- `models.py` defines the structured investigation domain model from Step 1.
- `adapters.py` normalizes incident, raw event, alert, correlation, MITRE and timeline context into engine input.
- `prompts.py` centralizes prompt text for structured investigation generation.
- `engine.py` generates validated `InvestigationBrief` objects and provides deterministic fallback behavior.
- `validators.py` enforces governance rules for confidence, unsupported claims, hypotheses and recommended actions.

No business logic is added to `api.py`.

## Engine Responsibilities

The engine:

- accepts incident and evidence context
- derives evidence references from available data
- produces hypotheses, findings, checks, actions, limitations and next steps
- supports LLM-assisted generation through an injected client
- validates generated output before returning it
- downgrades unsupported claims to speculative decision-support language
- keeps all recommended actions non-executable
- falls back to deterministic structured output when AI output is unavailable or invalid

## Relationship With Existing AI Analysis

Existing AI Analysis and Command Room-style views remain unchanged. The investigation engine is a backend foundation that can later be connected to those workflows through dedicated adapters or routers.

This step does not replace existing incident AI analysis. It creates a structured format that future steps can persist, render and iterate on.

## Deterministic Fallback

Fallback is mandatory. If an LLM call times out, returns malformed JSON, produces unsafe output or fails validation, the engine returns a valid deterministic `InvestigationBrief`.

Fallback output remains:

- structured
- evidence-aware
- professional
- explicit about limitations
- safe for analyst review

No traceback or raw runtime error is exposed in the brief.

## Explainability and Governance

The model uses claim classification:

- `EVIDENCE_BACKED`
- `INFERRED`
- `SPECULATIVE`
- `UNSUPPORTED`

Unsupported findings are not promoted as hard facts. Operational recommendations must require approval, and `execution_supported` remains `false`.

## Step 2 Limitations

Step 2 does not implement:

- remediation execution
- SSH, shell, firewall or active response actions
- autonomous investigation loops
- persistence for investigation sessions
- a new API endpoint
- frontend rendering
- cross-incident campaign intelligence

## Future Steps Enabled

This foundation enables later v0.6 work:

- persistent investigation sessions
- iterative evidence expansion
- analyst feedback loops
- confidence evolution over time
- structured investigation rendering in Incident Detail
- human-approved remediation planning

No remediation execution exists in Step 2.
