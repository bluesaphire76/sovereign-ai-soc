# v0.6 Step 12 - Remediation Governance Workflow

## Purpose

Step 12 connects the v0.6 remediation governance components into one coherent end-to-end flow.

Step 12 integrates remediation governance end-to-end but does not enable unrestricted production execution.

The workflow is:

```text
AI proposes.
Human approves.
Dry-run explains.
Readiness is assessed.
Policy validates.
Mock/no-op dispatch proves governance.
Production execution remains disabled unless future safe executors are explicitly implemented.
```

## Architecture

The implementation adds `remediation.workflow`, a deterministic orchestration layer that connects existing modules:

- remediation plan and action metadata
- approval records
- dry-run simulation
- rollback readiness
- execution readiness
- executor whitelist and policy evaluation
- mock/no-op dispatch
- execution audit and chain-of-custody metadata
- governance timeline and workflow state

No router, frontend control, migration or `api.py` business logic is added in this step.

## Workflow State Machine

The workflow exposes structured statuses:

- `PLAN_AVAILABLE`
- `AWAITING_APPROVAL`
- `APPROVED`
- `REJECTED`
- `DRY_RUN_READY`
- `READINESS_READY`
- `BLOCKED_BY_POLICY`
- `BLOCKED_BY_ROLLBACK`
- `READY_FOR_MOCK_DISPATCH`
- `MOCK_DISPATCH_COMPLETED`
- `NOOP_DISPATCH_COMPLETED`
- `EXECUTION_DISABLED`
- `FAILED_VALIDATION`

Every blocked state includes one or more blockers with a step, reason, severity and source.

## Governance Gates

The workflow enforces the remediation chain in order:

1. A remediation plan must exist.
2. The requested action must exist.
3. Forbidden actions are blocked.
4. Required human approval must be present and valid.
5. Dry-run output must be available before readiness review.
6. Rollback readiness must be acceptable for governed actions.
7. Execution readiness must return `READY_FOR_FUTURE_EXECUTOR`.
8. Executor policy must pass whitelist, approval, dry-run, readiness and rollback checks.
9. Dispatch is allowed only in `MOCK` or `NOOP` mode.
10. Dispatch records metadata only and does not mutate state.

No workflow path accepts arbitrary commands, shell payloads, unrestricted SSH or production-impact flags.

## Governance Timeline

The workflow creates a replay-friendly governance timeline with entries such as:

- plan available
- approval recorded or missing
- dry-run generated
- rollback readiness checked
- execution readiness assessed
- executor policy evaluated
- mock/no-op dispatch recorded
- audit metadata prepared

The timeline is designed for future UI and persistence integration. It is deterministic for the same plan/action inputs.

## Mock/No-Op Dispatch

Mock/no-op dispatch is used to validate governance flow, not to change target systems.

Dispatch results always preserve:

- `state_mutated = false`
- `production_impact = false`
- `execution_supported = false`

The workflow does not add an execute endpoint and does not expose a production execution path.

## Audit Wording

Audit event objects use non-misleading governance wording:

- `REMEDIATION_GOVERNANCE_STATE_EVALUATED`
- `REMEDIATION_WORKFLOW_ADVANCED`
- `REMEDIATION_WORKFLOW_BLOCKED_BY_POLICY`
- `REMEDIATION_WORKFLOW_BLOCKED_BY_ROLLBACK`
- `REMEDIATION_WORKFLOW_BLOCKED_BY_MISSING_APPROVAL`
- `REMEDIATION_MOCK_DISPATCH_COMPLETED`
- `REMEDIATION_NOOP_DISPATCH_COMPLETED`

The workflow deliberately avoids event names such as `remediation_executed`.

## Frontend and API

Step 12 does not add frontend UI or backend endpoints.

Future UI should render the workflow state, blockers and governance timeline in the existing Incident Detail / Command Room style. Allowed labels should remain governance-oriented, such as:

- Generate dry run
- Check readiness
- Check policy
- Mock dispatch
- View governance timeline

Future UI must not use labels such as:

- Execute
- Run remediation
- Apply fix
- Remediate now

## Limitations

Step 12 is not a SOAR workflow and not an active response layer.

It does not implement:

- arbitrary shell execution
- unrestricted SSH execution
- real firewall changes
- service stop or restart
- process kill
- file quarantine
- host isolation
- Wazuh active response execution
- production configuration changes

The value of this step is workflow coherence and safety: it proves the approval, dry-run, readiness, policy, whitelist and mock/no-op dispatch chain without enabling production remediation.

## Future Production Executor Prerequisites

Before any future real executor is considered, the product must add:

- explicit production executor allowlists per action type
- persistent approval/readiness/audit records
- RBAC-protected endpoints with no free-form command payloads
- rollback execution support and rollback evidence capture
- environment-level safety configuration
- operational owner approval policy
- UI that clearly separates mock/no-op governance from real execution
- documented target-system adapters with deterministic failure handling
