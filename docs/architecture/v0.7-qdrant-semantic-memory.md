# v0.7 Qdrant Semantic Memory

## Purpose

Qdrant is the local semantic retrieval layer for analyst support. It stores
derived, rebuildable memory from:

- SOC knowledge-base documents and incident-specific playbooks;
- historical incident summaries;
- governed Detection Control records;
- final or explicitly approved Case Closure records.

It enriches investigation and detection workflows without replacing the
PostgreSQL operational source of truth.

## Decision Boundary

Qdrant can retrieve context. It cannot make operational decisions.

Semantic memory must not:

- perform primary incident deduplication;
- set final severity or classification;
- apply noise suppression or exceptions;
- approve or apply Detection Control changes;
- close incidents or cases;
- approve remediation;
- replace deterministic correlation rules;
- treat historical similarity as proof of the same root cause or outcome.

RBAC, deterministic evidence, lifecycle rules, audit and human validation remain
authoritative.

## Source Types

| Source type | Content | Governance |
|---|---|---|
| `knowledge_base` | Playbooks, procedures and selected SOC guidance | Rebuilt from tracked Markdown; templates, archives and README files are excluded. |
| `historical_incident` | Redacted incident summaries and outcome context | Advisory only; open-incident inclusion is configurable. |
| `detection_control` | Detection rule, exception and suppression context | Cannot approve or apply a control. |
| `case_closure` | Final or human-approved closure decisions | Draft/open closure decisions are excluded. |

## Knowledge-Base Indexing

The general rebuild command is:

```bash
PYTHONPATH=. .venv/bin/python rag_index.py --recreate
```

For playbook-only maintenance, prefer the selective dry-run/apply path:

```bash
PYTHONPATH=. .venv/bin/python scripts/reindex_qdrant_playbooks.py --dry-run
PYTHONPATH=. .venv/bin/python scripts/reindex_qdrant_playbooks.py --apply
PYTHONPATH=. .venv/bin/python scripts/validate_qdrant_playbook_expansion.py
```

Selective playbook apply removes and replaces only playbook points. It
preserves historical incident, Detection Control and Case Closure memory.

Playbooks use YAML front matter and section-aware chunking. Metadata includes
domain, source, incident types, severity hints, MITRE data, applicability,
not-applicable conditions, target pages and tags. See
[Qdrant playbook metadata indexing](v0.7.0-qdrant-playbook-metadata-indexing.md)
and [the playbook authoring guide](../../knowledge_base/playbooks/README.md).

## Historical and Operational Memory

Historical incident backfill:

```bash
PYTHONPATH=. .venv/bin/python scripts/index_historical_incidents_to_qdrant.py --dry-run
PYTHONPATH=. .venv/bin/python scripts/index_historical_incidents_to_qdrant.py --apply
```

Detection Control and Case Closure backfill:

```bash
PYTHONPATH=. .venv/bin/python scripts/index_detection_case_memory_to_qdrant.py --dry-run
PYTHONPATH=. .venv/bin/python scripts/index_detection_case_memory_to_qdrant.py --apply
```

Historical retention is dry-run first:

```bash
PYTHONPATH=. .venv/bin/python scripts/qdrant_memory_retention.py --dry-run
PYTHONPATH=. .venv/bin/python scripts/qdrant_memory_retention.py --apply
```

Retention cleanup targets only `historical_incident`. It never deletes the
knowledge base. Detection/case backfill can prune stale records in those two
source types after a complete apply scan.

## Automatic Freshness

Best-effort automatic indexing is triggered by eligible incident, Detection
Control and Case Closure writes. It can run asynchronously and records state
under `.runtime/`.

Automatic indexing failure is visible but does not roll back the original SOC
write. This keeps Qdrant advisory and prevents vector infrastructure from
becoming an operational transaction dependency.

Optional systemd units provide periodic backfill and dry-run retention:

```text
deploy/systemd/ai-soc-qdrant-historical-backfill.service
deploy/systemd/ai-soc-qdrant-historical-backfill.timer
deploy/systemd/ai-soc-qdrant-detection-case-backfill.service
deploy/systemd/ai-soc-qdrant-detection-case-backfill.timer
deploy/systemd/ai-soc-qdrant-retention-dry-run.service
deploy/systemd/ai-soc-qdrant-retention-dry-run.timer
```

The scheduled retention unit is dry-run only.

```text
POST /semantic-memory/retention-cleanup
```

Workflow integrations:

```text
GET  /incidents/{incident_id}/similar-incidents
GET  /incidents/{incident_id}/recommended-playbooks
GET  /cases/{case_id}/recommended-playbooks
GET  /cases/{case_id}/closure/semantic-context
POST /detection-quality/semantic-context
POST /detection-control/semantic-context
```

ADMIN and ANALYST can inspect/search semantic memory and use semantic workflow
context. Operational backfill/retention endpoints are ADMIN-only and require
explicit confirmation for apply mode. VIEWER cannot access semantic-memory
search or workflow retrieval endpoints.

## Product Integrations

### Incident AI Brief and Case AI Analysis

Retrieved context is formatted as an explicitly advisory block with source,
score and excerpt. Prompts forbid using it as primary evidence or as a source
for severity, suppression, closure or correlation decisions.

### Similar Incidents

Historical matches help identify validation patterns. A match is not a
duplicate declaration and does not prove the same root cause.

### Recommended Playbooks

Incident and case detail retrieve playbook sections from `knowledge_base`.
Deterministic ranking uses metadata, authoritative incident/case facts and
platform/type compatibility before optional LLM synthesis.

The LLM can explain only retrieved titles. Invalid output, unknown titles,
timeouts or provider failure fall back to deterministic structured guidance.
See [Recommended Playbooks LLM synthesis](v0.7.0-qdrant-recommended-playbooks-llm.md).

### Detection Quality and Detection Control

Semantic context can surface prior tuning, exceptions, closures, historical
incidents and procedures. It cannot create, approve or apply a rule,
suppression or exception.

### Case Closure

Closure context includes similar approved/final closures, Detection Control
memory, historical incidents and policies. Deterministic closure readiness and
human approval remain authoritative.

## UI and Observability

The `Semantic Memory` settings page provides:

- capabilities and guardrails;
- Qdrant health, collection and index status;
- source-type counts and indexed-source visibility;
- auto-index status and recent events;
- read-only semantic search;
- ADMIN-only dry-run/apply controls;
- copyable CLI runbooks.

The Health page reports Qdrant reachability, configured collection existence,
point count and collection state. A reachable but missing/empty collection is
`WARN`, not healthy feature operation.

Grafana includes a dedicated Qdrant semantic-memory dashboard based on Qdrant
Prometheus metrics.
