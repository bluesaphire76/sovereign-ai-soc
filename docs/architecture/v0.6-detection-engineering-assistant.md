# v0.6 Detection Engineering Assistant Foundation

## Purpose

Step 7 introduces the foundation for an AI-assisted Detection Engineering Assistant.

The assistant analyzes available alert, incident, aggregate, suppression and historical investigation context to produce structured, analyst-reviewable detection engineering recommendations.

It is a governance and recommendation layer. It does not modify production detection rules.

## Architecture

The implementation lives in `detection_engineering/`:

- `models.py` defines typed findings, rule assessments, gaps, suppression candidates, tuning recommendations, correlation opportunities and reports.
- `scoring.py` provides deterministic scoring for noise, recurrence, confidence, quality impact, expected benefit and operational risk.
- `analyzer.py` normalizes detection context and derives findings from available data.
- `recommendations.py` converts findings into governed recommendations.
- `validators.py` checks evidence and approval boundaries.

No business logic is added to `api.py`.

No frontend changes are introduced in Step 7.

## Detection Engineering Categories

The foundation supports these recommendation categories:

- `NOISE_REDUCTION`
- `THRESHOLD_TUNING`
- `SUPPRESSION_CANDIDATE`
- `CORRELATION_IMPROVEMENT`
- `DETECTION_GAP`
- `MITRE_ENRICHMENT`
- `RULE_DEDUPLICATION`
- `RULE_QUALITY`
- `FALSE_POSITIVE_REDUCTION`
- `COVERAGE_IMPROVEMENT`

## Evidence-Backed Recommendation Model

Recommendations include:

- recommendation id
- title and description
- category
- severity
- confidence
- supporting evidence
- rationale
- expected benefit
- operational risk
- implementation notes
- validation suggestion
- rollback considerations
- approval requirement
- review status

Unsupported or evidence-light recommendations are downgraded to speculative confidence.

## Scoring Philosophy

Scoring is deterministic and conservative.

The assistant considers signals such as:

- repeated alert volume
- incident conversion rate
- suppression outcomes
- false-positive status where available
- low-severity recurrence
- MITRE metadata presence
- repeated rule identifiers
- cross-incident recurring context where supplied

There is no black-box machine-learning model in Step 7.

## Human Approval Requirement

All implementation-impacting recommendations require analyst or administrative review before action.

This includes:

- suppression candidates
- threshold tuning
- noise-reduction changes
- correlation logic changes
- rule deduplication
- false-positive reduction policies

The assistant proposes detection engineering improvements but does not modify production detection rules.

All implementation-impacting recommendations require human review and approval.

## Relationship With Detection Quality

Detection Quality measures synthetic scenario coverage and operational detection posture.

The Detection Engineering Assistant is the backend foundation for turning those signals and historical observations into governed detection improvement proposals.

Step 7 does not add a UI section to Detection Quality. A future step can render the recommendations in the existing enterprise Detection Quality page without changing its visual identity.

## Relationship With Cross-Incident Intelligence

Step 6 introduced explainable historical similarity and recurring pattern context.

Step 7 can accept historical context as input and treat recurring patterns as supporting context for detection engineering review. That context remains advisory and does not prove common cause or justify automatic detection changes.

## Governance Boundaries

Step 7 does not implement:

- automatic Wazuh rule changes
- automatic Suricata rule changes
- automatic suppression activation
- automatic threshold changes
- active response
- SOAR execution
- external threat intelligence feeds
- autonomous detection engineering
- frontend redesign

The output is structured decision support for detection engineers and SOC analysts.

## Limitations

Recommendations are only as complete as the supplied telemetry and historical context.

The assistant should be used to prioritize review, not to bypass validation. Proposed changes should be tested against representative historical events, synthetic scenarios and analyst expectations before production rollout.
