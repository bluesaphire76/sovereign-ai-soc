# v0.4 — DB Retention, Cleanup, Backup and Restore

## Purpose

This document defines the operational retention and backup/restore baseline for the Sovereign AI SOC local PostgreSQL runtime.

The goal is to reduce uncontrolled growth of raw operational data while preserving investigation, audit and case records.

## Runtime baseline

Current local runtime:

- PostgreSQL runs in Docker container `postgres-soc`
- Database user: `soc`
- Database name: `aisoc`
- Application services:
  - `ai-soc-api`
  - `ai-soc-frontend`
  - `ai-soc-worker`

## Protected data

The retention cleanup script does **not** delete investigation or audit records by default.

Protected tables:

- `incidents`
- `incident_cases`
- `case_incidents`
- `incident_audit`
- `incident_notes`
- `case_audit`
- `case_actions`
- `case_closure_checklists`
- `case_ai_analyses`
- `security_audit_events`
- `app_users`
- `wazuh_ingest_watermarks`
- `worker_heartbeats`

## Retention policy

Default policy:

| Table | Retention | Cleanup behavior |
|---|---:|---|
| `raw_events` | 90 days | Delete only rows older than cutoff and not linked to an incident |
| `security_alerts` | 180 days | Delete only rows older than cutoff and not linked to an incident |
| `event_aggregates` | 90 days | Delete aggregates inactive beyond cutoff |
| `incidents` | Not deleted | Protected |
| `incident_cases` | Not deleted | Protected |
| Audit tables | Not deleted | Protected |

## Configuration

Configured through environment variables:

```env
RETENTION_RAW_EVENTS_DAYS=90
RETENTION_SECURITY_ALERTS_DAYS=180
RETENTION_EVENT_AGGREGATES_DAYS=90
RETENTION_MAX_DELETE_ROWS=10000
RETENTION_REPORT_DIR=reports/retention
```

## Retention cleanup script

Script:

```bash
scripts/db_retention_cleanup.py
```

Dry-run is the default and safe mode:

```bash
cd ~/lab/ai-soc-assistant
source .venv/bin/activate

python scripts/db_retention_cleanup.py
```

The script writes a JSON report under:

```text
reports/retention/
```

Apply mode requires an explicit flag:

```bash
python scripts/db_retention_cleanup.py --apply
```

If candidate rows exceed `RETENTION_MAX_DELETE_ROWS`, apply mode will stop unless `--force` is explicitly provided.

Use `--force` only after manual review of the dry-run report.

```bash
python scripts/db_retention_cleanup.py --apply --force
```

## Backup before cleanup

Always create a backup before any apply-mode cleanup.

```bash
cd ~/lab/ai-soc-assistant

mkdir -p backups

docker exec postgres-soc pg_dump -U soc aisoc \
  > "backups/ai_soc_before_retention_cleanup_$(date -u +%Y%m%dT%H%M%SZ).sql"

latest_backup=$(ls -1t backups/ai_soc_before_retention_cleanup_*.sql | head -1)

echo "$latest_backup"
ls -lh "$latest_backup"
head -5 "$latest_backup"
```

Expected header:

```text
-- PostgreSQL database dump
```

The backup file must not be zero bytes.

## Restore test into a separate database

Use this for validation without destroying the active database.

```bash
latest_backup=$(ls -1t backups/ai_soc_before_retention_cleanup_*.sql | head -1)

docker exec postgres-soc dropdb -U soc --if-exists aisoc_restore_test
docker exec postgres-soc createdb -U soc aisoc_restore_test

docker exec -i postgres-soc psql -U soc -d aisoc_restore_test < "$latest_backup"

docker exec -i postgres-soc psql -U soc -d aisoc_restore_test <<'SQL'
select 'incidents' as table_name, count(*) from incidents
union all
select 'raw_events', count(*) from raw_events
union all
select 'security_alerts', count(*) from security_alerts
union all
select 'incident_cases', count(*) from incident_cases;
SQL
```

Drop the restore test database after validation:

```bash
docker exec postgres-soc dropdb -U soc --if-exists aisoc_restore_test
```

## Destructive restore to the active database

Use only when intentionally restoring the local runtime database.

1. Stop application services:

```bash
sudo systemctl stop ai-soc-worker
sudo systemctl stop ai-soc-api
sudo systemctl stop ai-soc-frontend
```

2. Create a final safety backup of the current database:

```bash
docker exec postgres-soc pg_dump -U soc aisoc \
  > "backups/ai_soc_before_destructive_restore_$(date -u +%Y%m%dT%H%M%SZ).sql"
```

3. Drop and recreate the target database:

```bash
docker exec postgres-soc dropdb -U soc --if-exists aisoc
docker exec postgres-soc createdb -U soc aisoc
```

4. Restore from backup:

```bash
latest_backup="backups/<backup-file>.sql"

docker exec -i postgres-soc psql -U soc -d aisoc < "$latest_backup"
```

5. Restart services:

```bash
sudo systemctl start ai-soc-api
sudo systemctl start ai-soc-frontend
sudo systemctl start ai-soc-worker
```

6. Validate:

```bash
curl -s http://127.0.0.1:8008/health
sudo journalctl -u ai-soc-api -n 80 --no-pager
sudo journalctl -u ai-soc-worker -n 80 --no-pager
```

## Operational notes

- Do not commit backup files.
- Do not commit generated validation PDFs.
- Keep `backups/` ignored by Git.
- Keep `reports/validation/*.pdf` ignored by Git.
- Run dry-run first, review the report, then run apply only when the candidate rows are expected.
- Raw/security rows linked to incidents are preserved by the cleanup script.
