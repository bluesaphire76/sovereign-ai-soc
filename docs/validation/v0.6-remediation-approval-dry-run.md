# v0.6 Step 9 - Remediation Approval and Dry-Run Foundation

## Purpose

Step 9 adds the governance layer around AI-proposed remediation plans. The platform can now model human approval decisions and generate deterministic dry-run output for remediation plans and actions.

This is a planning and governance foundation only.

Step 9 does not execute remediation actions.

Approval records prepare future execution governance but do not trigger execution in this step.

## Architecture

The remediation workflow is split into safe domain modules:

- `remediation.approvals` defines approval actors, requests, records, decisions, status and role-based policy evaluation.
- `remediation.dry_run` generates non-mutating simulation output from remediation plan metadata.
- `remediation.audit` creates audit-friendly event objects for approvals and dry-runs without writing unsafe details.
- `remediation.validators` validates approval records and dry-run results in addition to Step 8 plan/action validation.

No business logic is added to `api.py`.

No endpoint is added in Step 9. The domain layer is ready for a future dedicated remediation router if the product requires API exposure.

## Approval Workflow

The approval flow is designed for future persistence and UI integration:

1. AI or deterministic logic proposes a remediation plan.
2. Analyst or administrator reviews the plan.
3. The approval policy evaluates role, risk, approval requirement and forbidden status.
4. A decision record is produced: approve, reject, defer, request more evidence, or mark for admin review.
5. The record remains governance metadata only and does not trigger execution.

Supported approval statuses:

- `PENDING_REVIEW`
- `APPROVED`
- `REJECTED`
- `DEFERRED`
- `REQUIRES_ADMIN`
- `FORBIDDEN`
- `EXPIRED`
- `CANCELLED`

Supported decisions:

- `APPROVE`
- `REJECT`
- `DEFER`
- `REQUEST_MORE_EVIDENCE`
- `MARK_FOR_ADMIN_REVIEW`

## Role-Based Policy

Approval rules are deterministic and conservative:

| Role | Approval capability |
| --- | --- |
| `VIEWER` | Cannot approve remediation decisions |
| `ANALYST` | Can approve low/medium-risk non-destructive actions |
| `ADMIN` | Can approve admin-required governed actions |
| `SECURITY_LEAD` | Reserved for future high-risk governance models |

Forbidden-by-default actions cannot be approved in Step 9.

Approval requires a rationale. Records without rationale are invalid for governance purposes.

## Dry-Run Design

Dry-run is pure simulation from structured remediation metadata. It explains what would be reviewed and what would be required before a future execution layer could exist.

Dry-run output includes:

- action or plan summary
- target summary
- expected impact
- affected systems
- prerequisites
- pre-checks and post-checks
- rollback readiness
- risk summary
- blocked or missing conditions
- execution readiness status
- limitations

Dry-run does not:

- call shell commands
- contact targets
- modify systems
- write production configuration
- trigger Wazuh active response
- execute remediation

## Dry-Run Status

Supported dry-run statuses:

- `READY_FOR_REVIEW`
- `BLOCKED_BY_POLICY`
- `MISSING_APPROVAL`
- `MISSING_ROLLBACK`
- `MISSING_EVIDENCE`
- `FORBIDDEN`
- `NOT_SUPPORTED`

`NOT_SUPPORTED` is expected in Step 9 because no execution layer exists.

## Audit Events

The audit helper creates structured, sanitized event objects for:

- remediation approval requested
- remediation approved
- remediation rejected
- remediation deferred
- admin review requested
- forbidden approval attempted
- dry-run generated
- dry-run blocked by policy

Audit details intentionally omit executable command content and raw prompt-style data.

## Governance Boundaries

Step 9 preserves the product's human-in-the-loop remediation principle:

AI proposes.
Human reviews.
Dry-run explains.
Execution does not exist yet.

No approval record causes operational action. No dry-run result should be interpreted as execution readiness.

## Future Evolution

This foundation prepares later work for:

- remediation approval persistence
- remediation UI review panels
- execution readiness governance
- simulation reports
- rollback validation
- safe whitelisted executors
- execution audit trail

Those capabilities require dedicated future steps with explicit safety controls.
