# v0.7 Service Operations and Operation History

## Purpose

Service Operations adds governed status checks and restart controls for a
small allowlist of AI SOC services. Operation History provides a searchable,
auditable view of those checks, previews, attempts and outcomes.

This is not a general remote shell or arbitrary service manager.

## Allowlisted Services

| Service key | Runtime | Restart policy |
|---|---|---|
| `ai_soc_worker` | systemd | Governed restart supported |
| `ai_soc_api` | systemd | Status only; self-restart is blocked |
| `ai_soc_frontend` | systemd | Governed restart supported |
| `wazuh_manager` | Docker | Governed restart supported when the configured container exists |
| `suricata` | Docker | Governed restart supported when the configured container exists |

Unknown service keys are rejected.

## Governance Flow

1. Read the current service status.
2. Submit a restart preview with a required reason.
3. Review impact, runtime support, permission checks and warnings.
4. An ADMIN submits an explicit confirmed restart request.
5. The backend runs only the allowlisted command family.
6. Pre/post status, safe output and outcome are stored.
7. Security Audit and Operation History retain the governance record.

ADMIN and ANALYST can request previews. Only ADMIN can execute a restart.
VIEWER can read status and history.

Systemd restart execution uses non-interactive sudo and the committed
`deploy/sudoers/ai-soc-service-operations` allowlist. The API service cannot
restart itself through its own request path.

## API

```text
GET  /service-operations/services
GET  /service-operations/services/{service_key}/status
POST /service-operations/services/{service_key}/restart-preview
POST /service-operations/services/{service_key}/restart
GET  /service-operations/operations
GET  /service-operations/operations/{operation_id}
```

Operation History supports filters for service, operation type, status and
free-text search, with pagination up to 200 records per request.

## UI

- Detection Control Plane contains the Service Operations panel.
- System Information contains `Operation History`.
- History records show actor, reason, service, action, pre/post state, outcome,
  related configuration version and a redacted safe error/message.

## Safety Controls

- required reason;
- explicit confirmation;
- fixed service registry;
- fixed `systemctl` or `docker restart` command shape;
- timeout and output-size limits;
- secret-like output redaction;
- backend RBAC;
- security audit events;
- no shell interpolation and no arbitrary command input.
