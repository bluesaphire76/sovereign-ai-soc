# AI SOC v0.7.0 Exceptions and Noise Operations

v0.7 adds an operational UX for reviewing detection rules, exceptions and
noise suppression entries without bypassing the governed lifecycle.

## Scope

- Detection Control now includes an **Exceptions and Noise Operations** panel.
- The panel covers active and inactive detection rules, noise suppression entries and exceptions from lifecycle-managed items plus directly managed detection-control rules.
- Lifecycle-projected rules are de-duplicated so the lifecycle item remains the operational source of truth.
- Match evidence is read-only and based on recent local records from `raw_events`, `security_alerts`, `incidents` and `event_aggregates`.

## Backend endpoints

All endpoints are under the existing Detection Control router.

- `GET /detection-control/operations/overview`
- `GET /detection-control/operations/noise-suppression`
- `GET /detection-control/operations/exceptions`
- `GET /detection-control/operations/rules`
- `GET /detection-control/operations/items/{item_id}/summary`
- `GET /detection-control/operations/items/{item_id}/matched-events`
- `POST /detection-control/operations/match-preview`
- `POST /detection-control/operations/items/{item_id}/extend-review`
- `POST /detection-control/operations/items/{item_id}/mark-reviewed`

## Scope classification

Operational entries expose a scope risk classification:

- `narrow`: host, user, rule or similarly specific matcher.
- `moderate`: group, subnet, domain or network-level scope.
- `broad`: global scope or wildcard-like matcher.
- `dangerously_broad`: global scope combined with wildcard-only matching.
- `unknown`: insufficient scope information.

Broad, dangerously broad and unknown entries are automatically flagged for review.

## Counters and match evidence

The UI does not fake hit counters.

- Lifecycle entries use persisted `hit_count` and `last_hit_at` where available.
- Directly managed rules show hit counters as unavailable unless a persisted counter exists in a future schema.
- Event evidence and match preview scan recent local records with a capped scan window and return observed matches for analyst review.

## Review workflow

Review actions do not apply detection configuration, disable controls or restart services.

- `mark-reviewed` stores `operations_review` metadata and records audit.
- `extend-review` updates `expires_at` for lifecycle items, stores review metadata and records audit.
- Managed rules store review metadata in `metadata_json`.
- Lifecycle items also receive lifecycle history events for review activity.

## Audit events

This capability adds these security audit event types:

- `DETECTION_OPERATIONS_MATCH_PREVIEW_RUN`
- `DETECTION_OPERATIONS_REVIEW_MARKED`
- `DETECTION_OPERATIONS_REVIEW_EXTENDED`

Matcher values are not written to audit details; audits store matcher hash and length instead.
