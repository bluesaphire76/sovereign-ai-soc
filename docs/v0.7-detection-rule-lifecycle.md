# AI SOC v0.7 Detection Rule Lifecycle

AI SOC v0.7 introduces governed lifecycle management for detection rules, noise suppression rules and exceptions in the Detection Control Plane.

The lifecycle is focused on detection engineering governance. It is not a generic workflow engine.

## Policy Types

Lifecycle governance applies to:

- `DETECTION_RULE`
- `NOISE_SUPPRESSION`
- `EXCEPTION`

Each lifecycle item has a stable `rule_key`, an incrementing version number, content JSON, owner, business reason, source system, validation status and optional expiration/risk metadata.

## States

Supported states:

- `DRAFT`: editable working change.
- `PROPOSED`: submitted for admin review.
- `APPROVED`: approved but not active.
- `ACTIVE`: currently applied to the effective detection configuration.
- `DISABLED`: intentionally disabled through governed change.
- `SUPERSEDED`: replaced by a newer active item with the same rule key.
- `FAILED_VALIDATION`: blocked by deterministic validation.
- `ROLLED_BACK`: reserved for rollback linkage.
- `REJECTED`: rejected during review.

## Transition Model

Allowed transitions are enforced server-side:

- `DRAFT -> PROPOSED`
- `DRAFT -> FAILED_VALIDATION`
- `PROPOSED -> APPROVED`
- `PROPOSED -> REJECTED`
- `PROPOSED -> DRAFT`
- `PROPOSED -> FAILED_VALIDATION`
- `APPROVED -> ACTIVE`
- `APPROVED -> REJECTED`
- `ACTIVE -> DISABLED`
- `ACTIVE -> SUPERSEDED`
- `DISABLED -> DRAFT`
- `DISABLED -> PROPOSED`
- `FAILED_VALIDATION -> DRAFT`
- `FAILED_VALIDATION -> PROPOSED`
- `REJECTED -> DRAFT`
- `SUPERSEDED -> ROLLED_BACK`

Invalid transitions return a structured API error with the current state, requested target state and allowed transitions.

## RBAC

RBAC follows the existing AI SOC roles:

- `ADMIN`: create, edit, submit, approve, reject, apply, disable, clone, delete draft and view history.
- `ANALYST`: create drafts, edit own drafts, validate, submit, return to draft, clone and view history.
- `VIEWER`: read-only.

Approval, rejection, apply and disable are admin-only. Rejection and disable require a reason.

Denied attempts are audited.

## Validation

Validation is deterministic and does not execute external commands.

Common checks:

- valid policy type;
- title, owner and business reason;
- known source system;
- non-empty match criteria;
- no wildcard-only match;
- Detection Control Plane payload compatibility.

`NOISE_SUPPRESSION` checks:

- `action=suppress`;
- non-empty scope;
- wildcard host warning;
- no expiration/review date warning.

`EXCEPTION` checks:

- expiration date or explicit no-expiration justification;
- no expiration warning when justified;
- high/critical severity warning.

`DETECTION_RULE` checks:

- match criteria or expression;
- warning when severity/risk mapping is missing;
- warning when MITRE mapping or test scenario is missing.

Validation results are stored on the lifecycle item as `not_validated`, `passed` or `failed`.

## Audit And History

Every lifecycle transition writes:

- a lifecycle history event;
- a `SecurityAuditEvent`.

Lifecycle audit event names include:

- `DETECTION_RULE_DRAFT_CREATED`
- `DETECTION_RULE_UPDATED`
- `DETECTION_RULE_VALIDATED`
- `DETECTION_RULE_SUBMITTED`
- `DETECTION_RULE_APPROVED`
- `DETECTION_RULE_REJECTED`
- `DETECTION_RULE_APPLIED`
- `DETECTION_RULE_DISABLED`
- `DETECTION_RULE_SUPERSEDED`
- `DETECTION_RULE_CLONED`
- `DETECTION_RULE_TRANSITION_DENIED`

History is visible from the lifecycle detail view.

## Config Versioning Integration

Applying an approved lifecycle item uses the existing Step 2 configuration versioning path.

Apply behavior:

- validates the lifecycle item again;
- maps the lifecycle item into the matching config domain;
- updates the effective configuration payload;
- calls the existing config apply mechanism;
- creates a new active `DetectionConfigVersion`;
- stores `related_config_version_id` on the lifecycle item;
- marks older active lifecycle items with the same `rule_key` as `SUPERSEDED`.

Disable behavior also creates a new active config version with the rule disabled.

## Service Restart Integration

Apply and disable return:

- `restart_recommended`;
- `affected_services`;
- linked config version id.

The lifecycle does not restart services automatically. Operators use the existing Service Operations panel after apply or disable.

## Manual Validation

Admin flow:

1. Login as `ADMIN`.
2. Open Settings, Detection Control Plane.
3. Create a `NOISE_SUPPRESSION` draft.
4. Validate the draft.
5. Submit it for approval.
6. Approve it.
7. Apply it.
8. Confirm the item becomes `ACTIVE`.
9. Confirm a related config version is shown.
10. Confirm restart is recommended but not automatically executed.
11. Open lifecycle history and verify events.
12. Open Security Audit and verify lifecycle audit events.
13. Disable the active item with a mandatory reason.
14. Confirm the item becomes `DISABLED` and a new config version is linked.

Analyst flow:

1. Login as `ANALYST`.
2. Create a draft.
3. Validate and submit it.
4. Confirm approve, apply and disable are not available.

Viewer flow:

1. Login as `VIEWER`.
2. Open Detection Control Plane.
3. Confirm lifecycle items are visible read-only.
4. Confirm create and transition actions are not available.

## Operational Notes

Create the lifecycle tables in an existing deployment with:

```bash
python scripts/create_detection_rule_lifecycle_tables.py
```

Then restart the API and frontend services after deploying the backend and frontend changes.
