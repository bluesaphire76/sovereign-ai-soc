# v0.6 Step 11 - Controlled Executor Foundation

## Purpose

Step 11 introduces a whitelist-only controlled executor foundation for future remediation governance.

This step does not implement real remediation execution.

The framework exists to validate and route approved, rollback-aware remediation actions into NOOP or MOCK dispatch results only. It prepares the architecture for future safe executor work while preserving the product boundary:

```text
AI proposes.
Human approves.
Dry-run explains.
Readiness is assessed.
Only whitelisted NOOP/MOCK dispatch is available in Step 11.
Real execution still does not exist.
```

## Architecture

The executor package is intentionally separate from `api.py` and from the remediation planner:

- `executor.models` defines typed executor actions, targets, policy decisions and dispatch results.
- `executor.whitelist` defines the explicit action allowlist.
- `executor.validators` blocks arbitrary command content, raw shell parameters and production-impact flags.
- `executor.policy` validates approval, dry-run, rollback readiness, readiness status and whitelist membership.
- `executor.dispatcher` accepts only `ExecutorAction` plus a policy decision.
- `executor.noop_executor` records NOOP dispatch metadata only.
- `executor.mock_executor` records MOCK dispatch metadata only.
- `executor.audit` emits sanitized audit event objects.

No router or endpoint is added in this step.

## Whitelist-Only Model

Only explicitly modeled action types can be represented:

- `BLOCK_IP`
- `UNBLOCK_IP`
- `CREATE_TICKET`
- `NOTIFY_OWNER`
- `ESCALATE_CASE`
- `COLLECT_FORENSIC_EVIDENCE`

The whitelist constrains:

- allowed target types
- allowed executor modes
- approval requirement
- rollback readiness requirement
- production impact policy

All whitelist entries are restricted to `NOOP` and `MOCK` modes.

## Safety Boundaries

The executor framework forbids:

- arbitrary shell commands
- raw command parameters
- unrestricted SSH
- subprocess passthrough
- free-form execution
- production-impacting dispatch
- autonomous remediation

Executor actions force:

- `execution_supported = false`
- `production_impact_allowed = false`
- `command_preview_is_executable = false`

Validators reject:

- shell-like operators in command preview
- parameters such as `command`, `raw_command`, `shell`, `script`, `ssh` or `subprocess`
- disabled or missing whitelist entries
- target types outside the whitelist entry
- non-NOOP/MOCK modes

## Policy Engine

The policy engine requires:

1. A valid `ExecutorAction`.
2. A matching whitelist entry.
3. Approved remediation record when approval is required.
4. Matching dry-run reference.
5. Matching execution readiness assessment.
6. Acceptable rollback readiness when required.
7. NOOP or MOCK mode only.

When these checks pass, the result is still only:

- `ALLOWED_FOR_NOOP`
- `ALLOWED_FOR_MOCK`

That means dispatch metadata can be recorded. It does not mean production action is possible.

## Dispatcher

The dispatcher accepts:

- `ExecutorAction`
- `ExecutorPolicyDecision`

If validation or policy blocks the action, the dispatcher returns a blocked result.

If policy allows the action, the dispatcher routes to:

- `NoopExecutor`
- `MockExecutor`

Both return deterministic metadata with:

- `state_mutated = false`
- `production_impact = false`
- `execution_supported = false`

## Audit Trail

Executor audit events are sanitized and governance-oriented:

- `EXECUTOR_POLICY_EVALUATED`
- `EXECUTOR_ACTION_BLOCKED`
- `EXECUTOR_ACTION_DISPATCHED_NOOP`
- `EXECUTOR_ACTION_DISPATCHED_MOCK`
- `EXECUTOR_VALIDATION_FAILED`
- `EXECUTOR_UNSUPPORTED_REAL_EXECUTION_BLOCKED`

Audit details do not include command previews or executable material.

## Relationship With Step 8-10

The controlled executor foundation consumes artifacts from previous remediation steps:

- remediation action
- approval record
- dry-run result
- execution readiness assessment
- rollback readiness assessment

The adapter `executor_action_from_remediation` converts eligible remediation action types into executor actions. Unsupported remediation actions are rejected before policy evaluation.

## Non-Goals

Step 11 does not implement:

- real execution
- execute endpoint
- SSH execution
- Wazuh active response
- firewall changes
- user account changes
- service stop/restart
- file quarantine
- process kill
- host isolation
- SOAR automation
- frontend execution controls

## Future Work

Future versions may add a real executor only after:

- persisted approval and readiness records exist
- executor allowlist is reviewed by role and environment
- rollback execution is modeled and governed
- audit persistence is implemented
- RBAC routes are designed
- UI clearly distinguishes NOOP/MOCK/readiness from real execution
- target-specific adapters are safe, typed and non-free-form

Until then, the executor package is governance scaffolding only.
