# v0.6 — Remediation Replay Simulation

Step 19 adds a read-only replay layer for the remediation governance workflow.

The replay engine helps analysts understand how the current remediation scenario would move through:

- remediation plan generation
- AI governance safeguards
- approval gates
- dry-run simulation
- rollback readiness
- remediation audit trail preview
- final human decision posture

## What Replay Simulates

Replay composes existing remediation metadata and produces a structured timeline. It answers:

- which actions are proposed
- which approval gates apply
- which dry-run findings would appear
- whether rollback readiness is acceptable
- which governance warnings or policy blockers remain
- what audit trail preview would be produced

## Safety Boundary

Replay is not event injection and not remediation execution.

It does not:

- write to Wazuh or Suricata
- mutate incidents, cases, hosts, users, files, firewall rules or services
- execute remediation
- execute rollback
- call a shell or executor
- bypass human approval

All replay responses include:

- `execution_supported=false`
- `state_mutated=false`
- `replay_mode=READ_ONLY`
- `human_decision_required=true`

## UI

The Incident Command Room shows a compact `Replay Simulation` section. It displays the read-only replay mode, phase timeline, proposed actions, blockers, warnings and final recommendation.

## Limitations

Replay uses the current remediation intelligence and governance metadata. If one subsystem is unavailable, replay returns partial output with warnings instead of failing the Incident Detail page.
